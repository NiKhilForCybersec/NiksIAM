IAM Concepts & Fundamentals

Identity and Access Management (IAM) is the security discipline that ensures the right individuals access the right resources at the right times for the right reasons. This chapter covers the foundational concepts that underpin all modern IAM systems.

What is IAM and Why It Matters

Identity and Access Management encompasses the policies, processes, and technologies that enable organizations to manage digital identities and control user access to critical information. In today's threat landscape, IAM serves as the foundation of security architecture.

💡 Key Insight

According to Verizon's Data Breach Investigations Report, over 80% of breaches involve compromised credentials. Robust IAM is not optional—it's the first line of defense.

Core IAM Functions

  • Identity Management: Creating, maintaining, and deleting user accounts and attributes
  • Access Management: Controlling what resources users can access and what actions they can perform
  • Authentication: Verifying that users are who they claim to be
  • Authorization: Determining what authenticated users are allowed to do
  • Governance: Ensuring access is appropriate, reviewed, and compliant

Identity Lifecycle Management

The identity lifecycle describes the stages an identity goes through from creation to deletion. Effective lifecycle management ensures that access rights remain appropriate as users' relationships with the organization evolve.

Lifecycle Stages in Detail

Joiner Process (Onboarding) +

When a new employee, contractor, or partner joins the organization:

  • HR system triggers identity creation (authoritative source)
  • Identity is provisioned in directory services (AD, Entra ID)
  • Base access assigned based on role/department
  • Accounts created in downstream applications
  • Credentials issued (password, MFA enrollment)
  • Welcome communications sent with access instructions

Best Practice: Automate using HR-driven provisioning (Workday, SuccessFactors → IAM platform → Apps)

Mover Process (Role Changes) +

When an employee changes roles, departments, or responsibilities:

  • HR update triggers attribute changes
  • Old role-based access is removed
  • New role-based access is granted
  • Manager transfers are processed
  • Access review may be triggered

Challenge: Access accumulation—users often retain old access. Regular access reviews are essential.

Leaver Process (Offboarding) +

When an employee leaves the organization:

  • HR termination triggers immediate action
  • Account disabled (not deleted immediately for legal hold)
  • Active sessions terminated
  • MFA factors deregistered
  • Downstream app access revoked
  • Data ownership transferred
  • After retention period: account and data deleted

Critical: For involuntary terminations, disable access before the employee is notified.

Authentication Factors

Authentication verifies identity through one or more factors. Multi-factor authentication (MFA) combines factors from different categories to provide stronger security than any single factor alone.

Factor Type Description Examples Strength
Something You Know Knowledge-based secrets Password, PIN, Security Questions Low (phishable)
Something You Have Physical possession Phone, Hardware Token, Smart Card Medium
Something You Are Biometric characteristics Fingerprint, Face, Voice, Iris High
Somewhere You Are Location-based GPS, IP Address, Network Supplementary
Something You Do Behavioral patterns Typing rhythm, Mouse movement Supplementary
⚠️ Security Note

SMS-based OTP is considered the weakest MFA method due to SIM-swapping and SS7 vulnerabilities. Prefer authenticator apps, FIDO2 keys, or push notifications.

Authorization Models

Authorization determines what actions authenticated users can perform. Different models offer varying levels of flexibility and complexity.

Role-Based Access Control (RBAC)

Permissions are assigned to roles, and users are assigned to roles. This is the most common model in enterprise environments.

    ┌─────────┐        └───────────────┘     └─────────────────┘
    │   Bob   │───┐
    └─────────┘   │    ┌───────────────┐     ┌─────────────────┐
                  └───▶│  IT Admin     │────▶│ Manage Systems  │
    ┌─────────┐   ┌───▶│               │     │ Reset Passwords │

    └─────────┘                              └─────────────────┘

    ┌─────────┐        │  Finance      │────▶│ View Reports    │

Pros: Simple to understand, easy to audit, well-supported by most systems

Cons: Role explosion in large organizations, not context-aware

Attribute-Based Access Control (ABAC)

Access decisions based on attributes of the user, resource, action, and environment. Provides fine-grained, dynamic access control.

Relationship-Based Access Control (ReBAC)

Access determined by the relationship between users and resources. Popular in social platforms and document sharing systems.

    ReBAC MODEL
    ═══════════

    ┌─────────┐                              ┌──────────────┐
    │  Alice  │──────── owner ──────────────▶│  Document A  │
    └─────────┘                              └──────────────┘
         │                                          │
         │ manager_of                               │ parent
         ▼                                          ▼
    ┌─────────┐                              ┌──────────────┐
    │   Bob   │──────── viewer ─────────────▶│   Folder X   │
    └─────────┘                              └──────────────┘

    Access Rule: 
    user:Alice is owner of document:A → can edit
    user:Bob is viewer of folder:X → can read documents in X
    user:Alice is manager_of user:Bob → can see Bob's documents

Use Cases: Google Drive sharing, GitHub permissions, social networks

Policy-Based Access Control (PBAC)

Access governed by explicit policies defined in a policy language. Often uses standards like XACML or OPA (Open Policy Agent).

Rego (OPA Policy) 
# Allow read access to documents in user's department
allow {
    input.action == "read"
    input.resource.type == "document"
    input.resource.department == input.user.department
}

# Allow managers to approve requests from their reports
allow {
    input.action == "approve"
    input.resource.type == "request"
    input.resource.requester in data.reports[input.user.id]
}

# Deny access outside business hours for sensitive data
deny {
    input.resource.sensitivity == "high"
    not business_hours
}

business_hours {
    time.hour(time.now_ns()) >= 9
    time.hour(time.now_ns()) < 17
}

Single Sign-On (SSO)

SSO allows users to authenticate once and gain access to multiple applications without re-entering credentials. It improves user experience while centralizing authentication control.

SSO Benefits

  • User Experience: One login for all applications
  • Security: Fewer passwords to manage and potentially compromise
  • Reduced Helpdesk: Fewer password reset requests
  • Centralized Control: Single point for authentication policies and MFA
  • Audit Trail: Unified logging of authentication events

LDAP Protocol Deep Dive

Lightweight Directory Access Protocol (LDAP) is the standard protocol for accessing and managing directory information. It operates over TCP/IP (port 389 for LDAP, port 636 for LDAPS).

Common LDAP Operations

LDAP Query Examples 
# Search for all users in the domain
ldapsearch -x -H ldap://dc.corp.com -b "DC=corp,DC=com" "(objectClass=user)"

# Find a specific user by sAMAccountName
ldapsearch -x -H ldap://dc.corp.com -b "DC=corp,DC=com" \
    "(sAMAccountName=alice)" cn mail memberOf

# Find all members of a group
ldapsearch -x -H ldap://dc.corp.com -b "DC=corp,DC=com" \
    "(memberOf=CN=Admins,OU=Groups,DC=corp,DC=com)" cn

# Find users with password expiring in 7 days
ldapsearch -x -H ldap://dc.corp.com -b "DC=corp,DC=com" \
    "(&(objectClass=user)(pwdLastSet<=NOW-7d))" cn mail

# LDAP Filter Syntax:
# (&(condition1)(condition2))  - AND
# (|(condition1)(condition2))  - OR
# (!(condition))               - NOT
# (attribute=value)            - Equality
# (attribute=*value*)          - Substring
# (attribute>=value)           - Greater/Equal
# (attribute<=value)           - Less/Equal

Kerberos Authentication Flow

Kerberos is a network authentication protocol that uses tickets to prove identity. It's the default authentication protocol in Active Directory environments and provides mutual authentication between client and server.

    KERBEROS AUTHENTICATION FLOW
    ═════════════════════════════

    ┌──────────┐     ┌───────────────────────────────────────┐     ┌──────────┐
    │  Client  │     │         Key Distribution Center       │     │  Service │
    │  (User)  │     │    ┌──────────┐    ┌──────────┐      │     │  Server  │
    └────┬─────┘     │    │    AS    │    │   TGS    │      │     └────┬─────┘
         │           │    │(Auth Svc)│    │(Ticket   │      │          │
         │           │    │          │    │ Grant)   │      │          │
         │           │    └────┬─────┘    └────┬─────┘      │          │
         │           └─────────┼───────────────┼────────────┘          │
         │                     │               │                       │
    ═════╪═════════════════════╪═══════════════╪═══════════════════════╪═════
    STEP 1: Authentication Service Exchange                            │
         │                     │                                       │
         │  AS-REQ            │                                       │
         │  (User ID, TGS ID) │                                       │
         │────────────────────▶│                                       │
         │                     │                                       │
         │  AS-REP            │                                       │
         │  (TGT encrypted    │                                       │
         │   with user key)   │                                       │
         │◄────────────────────│                                       │
         │                     │                                       │
    ═════╪═════════════════════╪═══════════════╪═══════════════════════╪═════
    STEP 2: Ticket Granting Service Exchange   │                       │
         │                                     │                       │
         │  TGS-REQ                            │                       │
         │  (TGT + Service ID + Authenticator) │                       │
         │────────────────────────────────────▶│                       │
         │                                     │                       │
         │  TGS-REP                            │                       │
         │  (Service Ticket encrypted         │                       │
         │   with service key)                │                       │
         │◄────────────────────────────────────│                       │
         │                                                             │
    ═════╪═════════════════════════════════════════════════════════════╪═════
    STEP 3: Client/Server Exchange                                     │
         │                                                             │
         │  AP-REQ                                                     │
         │  (Service Ticket + Authenticator)                           │
         │─────────────────────────────────────────────────────────────▶
         │                                                             │
         │  AP-REP (optional, for mutual auth)                        │
         │◄─────────────────────────────────────────────────────────────
         │                                                             │
         │  ◄═══════════ SESSION ESTABLISHED ══════════▶               │
         │                                                             │

    KEY COMPONENTS:
    ───────────────
    AS (Authentication Service): Validates user credentials, issues TGT
    TGS (Ticket Granting Service): Issues service tickets
    TGT (Ticket Granting Ticket): Proves user authenticated to AS
    Service Ticket: Proves user authorized to access specific service
    Authenticator: Timestamp encrypted with session key (replay protection)
✓ Why Kerberos is Secure
  • Passwords never sent over the network
  • Mutual authentication between client and server
  • Tickets have limited lifetime (typically 10 hours for TGT)
  • Timestamps prevent replay attacks
  • Encryption protects ticket contents

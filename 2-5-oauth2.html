OAuth 2.0

ğŸ¯ What You'll Learn

OAuth 2.0 is about AUTHORIZATION (access to resources), NOT authentication. It's the protocol behind "Sign in with Google" and "Allow this app to access your calendar." This page makes it crystal clear.

ğŸš¨ The #1 Thing to Remember

OAuth is NOT Authentication!

OAuth 2.0 = AUTHORIZATION (what can you access)

OIDC = AUTHENTICATION (who are you) - built ON TOP of OAuth

Think of it this way:

  • OAuth: "Give this app permission to read my calendar"
  • OIDC: "Log me in using my Google account"

They're related but different! OIDC adds an ID Token for authentication.

ğŸš¶ Step-by-Step: "Calendly wants to read my Google Calendar"

Step 1: Click "Connect Google Calendar" in Calendly

Calendly redirects you to Google with: "I'm Calendly, I want calendar.read permission"

Step 2: Google Asks You

"Calendly wants to view your calendar. Allow?"

You click "Allow" (this is the CONSENT screen)

Step 3: Google Redirects Back with a Code

Google sends you back to Calendly with a short-lived authorization code

This code is like a claim ticket - it proves you said yes

Step 4: Calendly Exchanges Code for Token (Backend)

Calendly's server calls Google: "Here's the code + my client secret, give me a token"

Google responds with an Access Token (and maybe a Refresh Token)

Step 5: Calendly Uses the Token

Calendly calls Google Calendar API with the token in the header:

Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

Google Calendar returns your calendar data

Interview Tips

ğŸ’¡ How to Nail OAuth Questions

If asked "Explain OAuth 2.0":

"OAuth 2.0 is an authorization framework - it's about granting access to resources, not proving identity. A user can allow a third-party app to access their data without sharing their password. The app gets an access token that has specific scopes, like 'read calendar' but not 'delete calendar'. The token is short-lived for security."

If asked "What's the difference between OAuth and OIDC?":

"OAuth is about authorization - what can you access. OIDC is built on top of OAuth and adds authentication - who are you. OIDC adds the ID Token which contains user identity information. If you need login, use OIDC. If you just need API access, OAuth is enough."

If asked "Why was Implicit flow deprecated?":

"Implicit returned the access token directly in the URL fragment, making it visible in browser history and vulnerable to interception. Authorization Code with PKCE is more secure because the token exchange happens server-side, and PKCE prevents authorization code interception attacks."

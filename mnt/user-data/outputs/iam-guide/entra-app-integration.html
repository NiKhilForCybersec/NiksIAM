Application Integration

Entra ID supports integration with thousands of applications using SAML, OIDC, and other protocols. This chapter covers enterprise applications, app registrations, SSO configuration, and user provisioning.

Enterprise Applications

Enterprise applications are service principals representing applications users can access. They can be gallery apps, custom apps, or Microsoft apps.

Adding Gallery Applications

PowerShell - Add Gallery App 
# Search for gallery application
$template = Get-MgApplicationTemplate -Filter "displayName eq 'Salesforce'"

# Create enterprise app from gallery template
$params = @{
    DisplayName = "Salesforce - Production"
}
$app = New-MgApplicationTemplateInstantiate -ApplicationTemplateId $template.Id -BodyParameter $params

# The above creates both:
# - App Registration (Application object)
# - Enterprise App (Service Principal)

Write-Host "Application ID: $($app.Application.AppId)"
Write-Host "Service Principal ID: $($app.ServicePrincipal.Id)"

User and Group Assignment

PowerShell - Assign Users to App 
# Get service principal
$sp = Get-MgServicePrincipal -Filter "displayName eq 'Salesforce - Production'"

# Assign user to application
$params = @{
    PrincipalId = "user-object-id"
    ResourceId = $sp.Id
    AppRoleId = "00000000-0000-0000-0000-000000000000"  # Default role
}
New-MgUserAppRoleAssignment -UserId "user-object-id" -BodyParameter $params

# Assign group to application
$params = @{
    PrincipalId = "group-object-id"
    ResourceId = $sp.Id
    AppRoleId = "00000000-0000-0000-0000-000000000000"
}
New-MgGroupAppRoleAssignment -GroupId "group-object-id" -BodyParameter $params

# Require assignment (users must be assigned to access)
Update-MgServicePrincipal -ServicePrincipalId $sp.Id -AppRoleAssignmentRequired:$true

Terraform - App Registration 
# Create app registration for a web application
resource "azuread_application" "web_app" {
  display_name = "Contoso Web Application"
  
  # Who can sign in
  sign_in_audience = "AzureADMyOrg"  # Single tenant
  # Options: AzureADMyOrg, AzureADMultipleOrgs, AzureADandPersonalMicrosoftAccount
  
  # Web platform configuration
  web {
    homepage_url  = "https://app.contoso.com"
    redirect_uris = [
      "https://app.contoso.com/auth/callback",
      "https://app.contoso.com/signin-oidc"
    ]
    logout_url = "https://app.contoso.com/signout-callback-oidc"
    
    implicit_grant {
      access_token_issuance_enabled = false
      id_token_issuance_enabled     = false
    }
  }
  
  # API permissions
  required_resource_access {
    resource_app_id = "00000003-0000-0000-c000-000000000000"  # Microsoft Graph
    
    resource_access {
      id   = "e1fe6dd8-ba31-4d61-89e7-88639da4683d"  # User.Read
      type = "Scope"  # Delegated
    }
    resource_access {
      id   = "df021288-bdef-4463-88db-98f22de89214"  # User.Read.All
      type = "Role"   # Application
    }
  }
  
  # Optional claims in tokens
  optional_claims {
    id_token {
      name = "email"
    }
    id_token {
      name = "upn"
    }
  }
}

# Create service principal
resource "azuread_service_principal" "web_app" {
  client_id                    = azuread_application.web_app.client_id
  app_role_assignment_required = true
}

# Create client secret
resource "azuread_application_password" "web_app" {
  application_id = azuread_application.web_app.id
  display_name   = "App Secret"
  end_date       = "2025-12-31T00:00:00Z"
}

SSO Configuration

SAML SSO Setup

    SAML SSO CONFIGURATION
    ═══════════════════════

    ENTRA ID (IdP)                           APPLICATION (SP)
    ┌────────────────────┐                   ┌────────────────────┐
    │                    │                   │                    │
    │  SAML Settings:    │                   │  Required from     │
    │                    │                   │  Entra ID:         │
    │  • Entity ID       │──────────────────▶│  • IdP Entity ID   │
    │  • Login URL       │                   │  • Login URL       │
    │  • Logout URL      │                   │  • Certificate     │
    │  • Certificate     │                   │                    │
    │                    │                   │                    │
    │  Required from SP: │◀──────────────────│  SP Settings:      │
    │  • Entity ID       │                   │  • Entity ID       │
    │  • ACS URL         │                   │  • ACS URL         │
    │  • Attributes      │                   │  • Attribute map   │
    │                    │                   │                    │
    └────────────────────┘                   └────────────────────┘

    COMMON SAML ATTRIBUTES:
    ───────────────────────
    • NameID: user.userprincipalname (or user.mail)
    • email: user.mail
    • firstName: user.givenname
    • lastName: user.surname
    • displayName: user.displayname
    • groups: user.assignedroles (or group claims)
PowerShell - Configure SAML SSO 
# Get service principal
$sp = Get-MgServicePrincipal -Filter "displayName eq 'My SAML App'"

# Set SAML SSO mode
Update-MgServicePrincipal -ServicePrincipalId $sp.Id -PreferredSingleSignOnMode "saml"

# Configure SAML settings
$params = @{
    SingleSignOnSettings = @{
        SingleSignOnMode = "saml"
        RelayState = ""
    }
}

# Get SAML metadata URL
$metadataUrl = "https://login.microsoftonline.com/$tenantId/federationmetadata/2007-06/federationmetadata.xml?appid=$($sp.AppId)"

Write-Host "Metadata URL: $metadataUrl"
Write-Host "Entity ID: https://sts.windows.net/$tenantId/"
Write-Host "Login URL: https://login.microsoftonline.com/$tenantId/saml2"
Write-Host "Logout URL: https://login.microsoftonline.com/$tenantId/saml2"

User Provisioning (SCIM)

SCIM (System for Cross-domain Identity Management) enables automatic user provisioning and deprovisioning to applications.

    SCIM PROVISIONING FLOW
    ═══════════════════════

    ┌────────────────────┐                   ┌────────────────────┐
    │     ENTRA ID       │                   │    APPLICATION     │
    │  (SCIM Client)     │                   │   (SCIM Server)    │
    └─────────┬──────────┘                   └─────────┬──────────┘
              │                                        │
              │  1. Initial Sync (Full)                │
              │  ─────────────────────                 │
              │  GET /Users (fetch all)                │
              │────────────────────────────────────────▶│
              │                                        │
              │◀────────────────────────────────────────│
              │  User list response                    │
              │                                        │
              │  POST /Users (create missing)          │
              │────────────────────────────────────────▶│
              │                                        │
              │  2. Incremental Sync (every 40 min)    │
              │  ────────────────────────────────────  │
              │                                        │
              │  POST /Users (new users)               │
              │────────────────────────────────────────▶│
              │                                        │
              │  PATCH /Users/{id} (updates)           │
              │────────────────────────────────────────▶│
              │                                        │
              │  DELETE /Users/{id} (soft delete)      │
              │────────────────────────────────────────▶│
              │                                        │

    SCIM ENDPOINTS:
    ────────────────
    Base URL: https://app.example.com/scim/v2

    • GET    /Users              - List users
    • GET    /Users/{id}         - Get user
    • POST   /Users              - Create user
    • PATCH  /Users/{id}         - Update user
    • DELETE /Users/{id}         - Delete user
    • GET    /Groups             - List groups
    • POST   /Groups             - Create group
💡 Provisioning Best Practices
  • Start with a pilot group before provisioning all users
  • Map attributes carefully - review attribute mappings
  • Test with "Provision on demand" before enabling
  • Monitor provisioning logs for errors
  • Set up alerts for provisioning failures

Application Proxy

Application Proxy provides secure remote access to on-premises web applications without VPN.

    APPLICATION PROXY ARCHITECTURE
    ═══════════════════════════════

    EXTERNAL                    MICROSOFT                    ON-PREMISES
    (Internet)                  CLOUD                        NETWORK
                                                             
    ┌──────────┐    HTTPS    ┌──────────────┐              ┌──────────────┐
    │  Remote  │────────────▶│   Entra ID   │              │   Internal   │
    │  User    │             │  App Proxy   │              │     Web      │
    │          │◀────────────│   Service    │              │     App      │
    └──────────┘             └──────────────┘              └──────────────┘
                                    │                             ▲
                                    │ Outbound                    │
                                    │ HTTPS only                  │
                                    │ (443)                       │
                                    ▼                             │
                             ┌──────────────┐                     │
                             │  Connector   │─────────────────────┘
                             │   Server     │
                             │ (On-premises)│
                             └──────────────┘

    KEY BENEFITS:
    ─────────────
    • No inbound firewall rules required
    • No VPN needed
    • Pre-authentication with Entra ID
    • Conditional Access policies apply
    • SSO to on-premises apps (Kerberos/header-based)
    
    REQUIREMENTS:
    ─────────────
    • Entra ID P1 or P2 license
    • Windows Server for connector
    • Connector must have outbound internet access

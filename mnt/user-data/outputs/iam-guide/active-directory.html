On-Premises Active Directory

Active Directory (AD) is Microsoft's directory service for Windows domain networks. It stores information about objects on the network and makes this information easy to find and use. This chapter covers AD architecture, configuration, security hardening, and common attack vectors.

Key Architecture Components

Domain Controllers +

Domain Controllers (DCs) are servers that respond to authentication requests and store AD data.

  • NTDS.DIT: The AD database file stored on each DC
  • SYSVOL: Shared folder containing GPO files and scripts
  • Global Catalog: Partial replica of all objects in the forest for cross-domain queries
  • Read-Only DC (RODC): DC with read-only copy of AD for branch offices

Best Practice: Deploy at least 2 DCs per domain for redundancy.

Forests and Domains +

Forest: The top-level security boundary containing one or more domains.

  • Single schema shared across all domains
  • Global Catalog spans entire forest
  • Transitive trusts between all domains
  • Enterprise Admins have full forest access

Domain: Administrative boundary with its own security policies.

  • Separate password and account policies
  • Domain Admins have full domain access
  • Replication boundary for domain partition
Trust Relationships +
Trust Type Direction Transitive Use Case
Parent-Child Two-way Yes Automatic between parent/child domains
Tree-Root Two-way Yes Automatic between tree roots
External One/Two-way No Trusts with domains in other forests
Forest One/Two-way Yes Trust between entire forests
Shortcut One/Two-way Yes Optimize auth between distant domains
Realm One/Two-way Yes/No Trust with non-Windows Kerberos realm

FSMO Roles (Flexible Single Master Operations)

FSMO roles are special domain controller roles that perform operations that can only be done by a single DC to prevent conflicts. There are 5 FSMO roles: 2 forest-wide and 3 domain-wide.

FSMO Management Commands

PowerShell 
# View current FSMO role holders
Get-ADForest | Select-Object SchemaMaster, DomainNamingMaster
Get-ADDomain | Select-Object PDCEmulator, RIDMaster, InfrastructureMaster

# Alternative: Query all FSMO roles
netdom query fsmo

# Transfer FSMO role (graceful - both DCs online)
Move-ADDirectoryServerOperationMasterRole -Identity "DC02" -OperationMasterRole PDCEmulator

# Seize FSMO role (forceful - original holder offline)
Move-ADDirectoryServerOperationMasterRole -Identity "DC02" -OperationMasterRole PDCEmulator -Force

# Transfer all roles to a single DC
Move-ADDirectoryServerOperationMasterRole -Identity "DC02" `
    -OperationMasterRole SchemaMaster, DomainNamingMaster, PDCEmulator, RIDMaster, InfrastructureMaster
⚠️ FSMO Seizure Warning

Never bring a DC back online after its FSMO role has been seized. The old role holder must be demoted or rebuilt before rejoining the domain.

Active Directory Objects

AD stores information as objects with attributes. Understanding object types and their relationships is essential for administration.

Users, Groups, and Computers

User Account Types

  • Domain Users: Regular user accounts for employees
  • Service Accounts: Accounts for running services (prefer gMSA)
  • Computer Accounts: Machine identities (auto-created when joining domain)
PowerShell - User Management 
# Create new user
New-ADUser -Name "Alice Smith" `
    -SamAccountName "asmith" `
    -UserPrincipalName "asmith@corp.com" `
    -Path "OU=Users,OU=Corp,DC=corp,DC=com" `
    -AccountPassword (ConvertTo-SecureString "P@ssw0rd!" -AsPlainText -Force) `
    -Enabled $true `
    -ChangePasswordAtLogon $true

# Get user properties
Get-ADUser -Identity "asmith" -Properties *

# Find users with password never expires
Get-ADUser -Filter {PasswordNeverExpires -eq $true} -Properties PasswordNeverExpires

# Disable user account
Disable-ADAccount -Identity "asmith"

# Find inactive users (90 days)
$threshold = (Get-Date).AddDays(-90)
Get-ADUser -Filter {LastLogonDate -lt $threshold} -Properties LastLogonDate

Group Types and Scopes

Scope Members From Permissions In Best Use
Domain Local Any domain Same domain only Resource permissions
Global Same domain only Any domain Organize users by role
Universal Any domain Any domain Forest-wide access

AGDLP Strategy: Accounts → Global → Domain Local → Permissions

PowerShell - Group Management 
# Create security group
New-ADGroup -Name "Finance_Users" `
    -GroupScope Global `
    -GroupCategory Security `
    -Path "OU=Groups,DC=corp,DC=com"

# Add members to group
Add-ADGroupMember -Identity "Finance_Users" -Members "asmith","bjones"

# Get group members
Get-ADGroupMember -Identity "Domain Admins" -Recursive

# Find nested group memberships for a user
Get-ADUser -Identity "asmith" -Properties MemberOf | 
    Select-Object -ExpandProperty MemberOf

Computer Objects

Computer accounts are created automatically when machines join the domain.

PowerShell - Computer Management 
# Find all computers
Get-ADComputer -Filter * -Properties OperatingSystem, LastLogonDate

# Find stale computer accounts (90 days)
$threshold = (Get-Date).AddDays(-90)
Get-ADComputer -Filter {LastLogonDate -lt $threshold} -Properties LastLogonDate

# Pre-stage computer account
New-ADComputer -Name "WS-FINANCE-01" `
    -Path "OU=Workstations,DC=corp,DC=com" `
    -Enabled $true

# Reset computer account
Reset-ComputerMachinePassword -Server DC01 -Credential (Get-Credential)

Organizational Units (OUs) and GPOs

OUs are containers for organizing objects and applying Group Policy.

PowerShell - OU Management 
# Create OU structure
New-ADOrganizationalUnit -Name "Corp" -Path "DC=corp,DC=com"
New-ADOrganizationalUnit -Name "Users" -Path "OU=Corp,DC=corp,DC=com"
New-ADOrganizationalUnit -Name "Computers" -Path "OU=Corp,DC=corp,DC=com"
New-ADOrganizationalUnit -Name "Servers" -Path "OU=Corp,DC=corp,DC=com"

# Protect OU from accidental deletion
Set-ADOrganizationalUnit -Identity "OU=Corp,DC=corp,DC=com" `
    -ProtectedFromAccidentalDeletion $true

# Get all OUs
Get-ADOrganizationalUnit -Filter * | Select-Object Name, DistinguishedName

# Get GPOs linked to an OU
Get-GPInheritance -Target "OU=Corp,DC=corp,DC=com"

Group Policy Deep Dive

Group Policy enables centralized management of user and computer settings. Understanding GPO processing order and precedence is critical for effective administration.

    GROUP POLICY PROCESSING ORDER (LSDOU)
    ═══════════════════════════════════════

    Processing Order (Last applied wins):

    ┌─────────────────────────────────────────────────────────────────────┐
    │  1. LOCAL POLICY                                                     │
    │     └── gpedit.msc on individual computer                           │
    │         (Lowest precedence - easily overridden)                     │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    ┌─────────────────────────────────────────────────────────────────────┐
    │  2. SITE POLICY                                                      │
    │     └── Linked to AD Sites                                          │
    │         (Applied based on computer's IP/subnet)                     │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    ┌─────────────────────────────────────────────────────────────────────┐
    │  3. DOMAIN POLICY                                                    │
    │     └── Linked to domain root                                       │
    │         (Default Domain Policy for password settings)               │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    ┌─────────────────────────────────────────────────────────────────────┐
    │  4. OU POLICIES (parent to child)                                   │
    │     └── OU=Corp                                                     │
    │         └── OU=Departments                                          │
    │             └── OU=Finance   ◄── Highest precedence                 │
    └─────────────────────────────────────────────────────────────────────┘

    MODIFIERS:
    ──────────
    • Block Inheritance: Stops policies from parent containers
    • Enforced (No Override): Forces policy even when blocked
    • Security Filtering: Applies based on user/group membership
    • WMI Filtering: Applies based on WMI query results

Common Security GPO Settings

Password Policy Settings +

Location: Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Password Policy

Setting Recommended Value Notes
Minimum password length 14 characters NIST recommends 8+ but longer is better
Password complexity Enabled Requires 3 of 4 character types
Maximum password age 0 (never) or 365 NIST no longer recommends rotation
Password history 24 passwords Prevents immediate reuse
Account Lockout Policy +

Location: Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Account Lockout Policy

Setting Recommended Value
Account lockout threshold 5-10 invalid attempts
Account lockout duration 30 minutes
Reset lockout counter after 30 minutes
Audit Policy Settings +

Location: Computer Configuration → Policies → Windows Settings → Security Settings → Advanced Audit Policy Configuration

Critical events to audit:

  • Account Logon: Credential Validation (Success/Failure)
  • Account Management: User/Group Account Management (Success/Failure)
  • Logon/Logoff: Logon, Special Logon, Account Lockout (Success/Failure)
  • Object Access: File System, Registry (as needed)
  • Policy Change: Audit Policy Change, Authentication Policy Change
  • Privilege Use: Sensitive Privilege Use (Success/Failure)
  • System: Security State Change, System Integrity
PowerShell - GPO Management 
# Create new GPO
New-GPO -Name "Security-Baseline-Workstations"

# Link GPO to OU
New-GPLink -Name "Security-Baseline-Workstations" `
    -Target "OU=Workstations,OU=Corp,DC=corp,DC=com"

# Get all GPOs
Get-GPO -All | Select-Object DisplayName, GpoStatus, CreationTime

# Get GPO report
Get-GPOReport -Name "Security-Baseline-Workstations" -ReportType HTML `
    -Path "C:\Reports\GPO-Report.html"

(Get-GPInheritance -Target "OU=Corp,DC=corp,DC=com").GpoLinks

# Force Group Policy update on remote computer
Invoke-GPUpdate -Computer "WS-01" -Force

# Backup all GPOs
Backup-GPO -All -Path "C:\GPO-Backups"

Installation and Configuration

Installing AD DS Role

PowerShell - New Forest Installation 
# Install AD DS role
Install-WindowsFeature AD-Domain-Services -IncludeManagementTools

# Promote to first DC in new forest
Install-ADDSForest `
    -DomainName "corp.com" `
    -DomainNetBIOSName "CORP" `
    -ForestMode "WinThreshold" `
    -DomainMode "WinThreshold" `
    -InstallDNS:$true `
    -DatabasePath "D:\NTDS" `
    -LogPath "D:\NTDS" `
    -SYSVOLPath "D:\SYSVOL" `
    -SafeModeAdministratorPassword (ConvertTo-SecureString "DSRMPassword!" -AsPlainText -Force) `
    -Force:$true

# Add additional DC to existing domain
Install-ADDSDomainController `
    -DomainName "corp.com" `
    -InstallDNS:$true `
    -Credential (Get-Credential) `
    -DatabasePath "D:\NTDS" `
    -LogPath "D:\NTDS" `
    -SYSVOLPath "D:\SYSVOL" `
    -SafeModeAdministratorPassword (ConvertTo-SecureString "DSRMPassword!" -AsPlainText -Force)

AD Security Hardening

Securing Active Directory is critical as it's often the primary target for attackers. Implement defense-in-depth with these key controls.

    TIERED ADMINISTRATION MODEL
    ════════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │  TIER 0 - Domain Controllers & Identity                             │
    │  ─────────────────────────────────────────────────────────────────  │
    │  Assets: DCs, AD DS, PKI, ADFS, Identity management                │
    │  Accounts: Domain Admins, Enterprise Admins, Schema Admins         │
    │  Access: Only from Tier 0 PAWs                                     │
    │  Rule: NEVER log into lower tier systems                           │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
                                    │ No credentials flow down
                                    ▼
    ┌─────────────────────────────────────────────────────────────────────┐
    │  TIER 1 - Servers & Applications                                    │
    │  ─────────────────────────────────────────────────────────────────  │
    │  Assets: Member servers, applications, databases                    │
    │  Accounts: Server admins, app admins                               │
    │  Access: Only from Tier 1 PAWs or jump servers                     │
    │  Rule: NEVER log into Tier 0 or workstations                       │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
                                    │ No credentials flow down
                                    ▼
    ┌─────────────────────────────────────────────────────────────────────┐
    │  TIER 2 - Workstations & Users                                      │
    │  ─────────────────────────────────────────────────────────────────  │
    │  Assets: Workstations, laptops, end user devices                   │
    │  Accounts: Helpdesk, workstation admins                            │
    │  Access: Standard admin tools                                      │
    │  Rule: NEVER use privileged credentials                            │
    └─────────────────────────────────────────────────────────────────────┘

Critical Security Controls

🛡️

Protected Users Group

Members cannot use NTLM, DES, or RC4. Kerberos TGTs limited to 4 hours. No credential caching.

🔒

Credential Guard

Virtualization-based security isolates credentials from the OS. Blocks pass-the-hash and credential theft.

🔑

LAPS

Local Administrator Password Solution randomizes and rotates local admin passwords stored in AD.

💻

PAW (Privileged Access Workstation)

Dedicated hardened workstations for administrative tasks. No email, web browsing, or general use.

PowerShell - Security Configuration 
# Add account to Protected Users group
Add-ADGroupMember -Identity "Protected Users" -Members "admin-tier0"

# Check Protected Users membership
Get-ADGroupMember -Identity "Protected Users"

# Enable Credential Guard via GPO
# Path: Computer Configuration → Administrative Templates → System → Device Guard
# Setting: Turn On Virtualization Based Security → Credential Guard Configuration

# Install LAPS
# 1. Extend AD schema (run on Schema Admin workstation)
Update-AdmPwdADSchema

# 2. Set permissions on OUs for LAPS
Set-AdmPwdComputerSelfPermission -OrgUnit "OU=Workstations,DC=corp,DC=com"
Set-AdmPwdReadPasswordPermission -OrgUnit "OU=Workstations,DC=corp,DC=com" `
    -AllowedPrincipals "Helpdesk"

# 3. Deploy via GPO (enable and configure expiration)

# Find admin accounts with SPNs (Kerberoasting targets)
Get-ADUser -Filter {ServicePrincipalName -ne "$null"} -Properties ServicePrincipalName |
    Where-Object {$_.MemberOf -match "Admin"}

Common AD Attacks and Mitigations

Attack Description Mitigation
Pass-the-Hash Attacker uses stolen NTLM hash to authenticate without knowing password Credential Guard, Protected Users, disable NTLM where possible
Kerberoasting Request service tickets for accounts with SPNs, crack offline Use gMSA, strong passwords on service accounts, monitor 4769 events
Golden Ticket Forge TGT using stolen KRBTGT hash for any user Rotate KRBTGT password twice, detect anomalous TGTs
DCSync Replicate password hashes from DC using replication protocols Limit Replicating Directory Changes permissions, monitor 4662 events
NTLM Relay Relay captured NTLM authentication to other services Enable SMB signing, EPA, disable NTLM, LDAP signing
AS-REP Roasting Request TGT for accounts without pre-auth, crack offline Enable Kerberos pre-authentication for all accounts
🚨 Critical: KRBTGT Account

The KRBTGT account is the most sensitive account in AD. If compromised, attackers can create Golden Tickets. Rotate the password twice (with a minimum 10-hour gap) after any suspected compromise. The password change will invalidate all existing Kerberos tickets.

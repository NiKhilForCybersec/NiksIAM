Microsoft Entra ID Fundamentals

Microsoft Entra ID (formerly Azure Active Directory) is Microsoft's cloud-based identity and access management service. It provides authentication and authorization for Microsoft 365, Azure, and thousands of SaaS applications.

Entra ID vs Active Directory

Understanding the differences between on-premises Active Directory and cloud-based Entra ID is crucial for architects and administrators.

Aspect Active Directory Microsoft Entra ID
Deployment On-premises (Domain Controllers) Cloud-native (Microsoft managed)
Structure Forests, Domains, OUs Flat structure (Tenants)
Protocols Kerberos, NTLM, LDAP SAML, OAuth 2.0, OpenID Connect
Query Language LDAP Microsoft Graph API, OData
Device Management Group Policy (GPO) Intune / MDM
Authentication Kerberos tickets OAuth tokens, SAML assertions
Trust Model Forest/Domain trusts B2B collaboration, federation
Administration AD Users & Computers, PowerShell Entra Portal, Graph API, PowerShell

Identity Types

Cloud-Only Users

Users created and managed entirely in Entra ID with no on-premises footprint.

  • Source of authority is Entra ID
  • Password managed in the cloud
  • No dependency on on-premises infrastructure
  • Best for cloud-first organizations
PowerShell - Create Cloud User 
# Connect to Microsoft Graph
Connect-MgGraph -Scopes "User.ReadWrite.All"

# Create new cloud user
$passwordProfile = @{
    Password = "TempP@ssw0rd!"
    ForceChangePasswordNextSignIn = $true
}

New-MgUser -DisplayName "Alice Smith" `
    -UserPrincipalName "alice@contoso.com" `
    -MailNickname "alice" `
    -PasswordProfile $passwordProfile `
    -AccountEnabled

# Using Azure CLI
az ad user create \
    --display-name "Alice Smith" \
    --user-principal-name "alice@contoso.com" \
    --password "TempP@ssw0rd!" \
    --force-change-password-next-sign-in true

Synchronized Users

Users synced from on-premises Active Directory via Entra Connect or Cloud Sync.

  • Source of authority is on-premises AD
  • Changes made on-premises sync to cloud
  • Password can be synced (PHS) or federated
  • Hybrid identity scenarios
💡 Identifying Synced Users

Synced users have onPremisesSyncEnabled = true and cannot be modified directly in Entra ID (except for cloud-only attributes like licenses).

Guest Users (B2B)

External users invited to collaborate in your tenant. They authenticate with their home tenant or identity provider.

  • UserType = "Guest"
  • Authenticate with their own credentials
  • Limited default permissions
  • Can be assigned to groups, apps, and roles
  • Supports email one-time passcode for any email
PowerShell - Invite Guest User 
# Invite guest user via Microsoft Graph
New-MgInvitation -InvitedUserEmailAddress "partner@external.com" `
    -InviteRedirectUrl "https://myapps.microsoft.com" `
    -SendInvitationMessage:$true `
    -InvitedUserDisplayName "Partner User"

# Bulk invite from CSV
Import-Csv "guests.csv" | ForEach-Object {
    New-MgInvitation -InvitedUserEmailAddress $_.Email `
        -InviteRedirectUrl "https://myapps.microsoft.com" `
        -SendInvitationMessage:$true
}

External Identities (B2C)

Customer-facing identity solution for consumer applications. Separate Entra ID B2C tenant required.

  • Designed for customer/consumer apps
  • Supports social identity providers (Google, Facebook, etc.)
  • Custom sign-up/sign-in experiences
  • Scales to millions of users
  • Separate from workforce identity

Device Identities

    DEVICE IDENTITY OPTIONS
    ════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    ENTRA ID REGISTERED                               │
    │   ─────────────────────────────────────────────────────────────────  │
    │   • Personal devices (BYOD)                                         │
    │   • User signs in with personal account + adds work account         │
    │   • Device NOT joined to any directory                              │
    │   • Minimal device management                                       │
    │   • SSO to cloud apps                                               │
    │   • Supported: Windows 10+, iOS, Android, macOS                     │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    ENTRA ID JOINED                                   │
    │   ─────────────────────────────────────────────────────────────────  │
    │   • Corporate devices (cloud-only orgs)                             │
    │   • Device joined ONLY to Entra ID                                  │
    │   • Sign in with Entra ID account                                   │
    │   • Full MDM management via Intune                                  │
    │   • SSO to cloud and on-prem apps (with config)                     │
    │   • Windows Hello for Business                                      │
    │   • Supported: Windows 10+ only                                     │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    HYBRID ENTRA ID JOINED                            │
    │   ─────────────────────────────────────────────────────────────────  │
    │   • Corporate devices (hybrid orgs)                                 │
    │   • Device joined to BOTH on-prem AD AND Entra ID                   │
    │   • Existing domain-joined devices                                  │
    │   • GPO + Intune management                                         │
    │   • SSO to both cloud and on-prem resources                         │
    │   • Requires Entra Connect device sync                              │
    │   • Supported: Windows 10+, Windows Server 2016+                    │
    └─────────────────────────────────────────────────────────────────────┘

    DECISION FLOW:
    ──────────────

    Is it a personal device?
        │
        ├── YES → Entra ID Registered
        │
        └── NO → Do you have on-premises AD?
                    │
                    ├── NO → Entra ID Joined
                    │
                    └── YES → Is it a new device?
                                │
                                ├── YES (cloud-first) → Entra ID Joined
                                │
                                └── NO (existing) → Hybrid Entra ID Joined
PowerShell - Device Management 
# Get all devices
Get-MgDevice -All | Select-Object DisplayName, DeviceId, TrustType, 
    ApproximateLastSignInDateTime

# Filter by join type
Get-MgDevice -Filter "trustType eq 'AzureAd'" # Entra ID Joined
Get-MgDevice -Filter "trustType eq 'ServerAd'" # Hybrid Joined
Get-MgDevice -Filter "trustType eq 'Workplace'" # Registered

# Find stale devices (90 days)
$threshold = (Get-Date).AddDays(-90).ToString("yyyy-MM-ddTHH:mm:ssZ")
Get-MgDevice -Filter "approximateLastSignInDateTime le $threshold"

# Check device status on Windows
dsregcmd /status

# Key outputs to check:
# AzureAdJoined: YES/NO
# DomainJoined: YES/NO  
# WorkplaceJoined: YES/NO

Groups in Entra ID

Security Groups

Used for managing access to resources. Can be assigned to applications, Conditional Access policies, and Azure resources.

  • Assigned or dynamic membership
  • Can be synced from on-premises AD
  • Can be role-assignable (for Entra ID roles)
  • No collaboration features (no mailbox, Teams, etc.)

Microsoft 365 Groups

Collaboration groups that provision shared resources including mailbox, SharePoint site, Teams, and Planner.

  • Created when you create a Team, SharePoint site, etc.
  • Has owners and members
  • Can be public or private
  • Expiration policies available
  • Cannot be role-assignable

Dynamic Groups (P1 Required)

Membership automatically managed based on user/device attribute rules.

Dynamic Membership Rules 
# All users in Sales department
(user.department -eq "Sales")

# All users with Manager title
(user.jobTitle -contains "Manager")

# All users in US or Canada
(user.country -eq "United States") or (user.country -eq "Canada")

# All full-time employees (custom attribute)
(user.extensionAttribute1 -eq "FTE")

# All Windows 10+ devices
(device.deviceOSType -eq "Windows") and (device.deviceOSVersion -startsWith "10")

# All Intune compliant devices
(device.deviceManagementAppId -eq "0000000a-0000-0000-c000-000000000000") 
    and (device.isCompliant -eq true)

# Exclude guest users
(user.userType -eq "Member")

# Complex rule with multiple conditions
(user.department -eq "Engineering") and (user.jobTitle -contains "Senior") 
    and (user.accountEnabled -eq true)
⚠️ Dynamic Group Limitations
  • Membership updates can take minutes to hours
  • Maximum 15 dynamic membership rules per tenant (for assigned users)
  • Cannot mix dynamic and assigned membership
  • Role-assignable groups cannot be dynamic

Administrative Units

Administrative Units (AUs) provide a way to restrict administrative scope within Entra ID. They enable delegated administration for specific subsets of users, groups, or devices.

    ADMINISTRATIVE UNITS
    ═════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                        ENTRA ID TENANT                               │
    │                                                                      │
    │   Global Admin ──── Full access to entire tenant                    │
    │                                                                      │
    │   ┌───────────────────────┐     ┌───────────────────────┐          │
    │   │   AU: North America   │     │   AU: Europe          │          │
    │   │                       │     │                       │          │
    │   │   ┌─────┐ ┌─────┐    │     │   ┌─────┐ ┌─────┐    │          │
    │   │   │Users│ │Groups│    │     │   │Users│ │Groups│    │          │
    │   │   │(NA) │ │(NA)  │    │     │   │(EU) │ │(EU)  │    │          │
    │   │   └─────┘ └─────┘    │     │   └─────┘ └─────┘    │          │
    │   │                       │     │                       │          │
    │   │   Scoped Admins:      │     │   Scoped Admins:      │          │
    │   │   • User Admin (NA)   │     │   • User Admin (EU)   │          │
    │   │   • Helpdesk (NA)     │     │   • Helpdesk (EU)     │          │
    │   └───────────────────────┘     └───────────────────────┘          │
    │                                                                      │
    └─────────────────────────────────────────────────────────────────────┘

    USE CASES:
    ──────────
    • Geographic delegation (NA, EMEA, APAC admins)
    • Departmental delegation (HR admins manage HR users)
    • Subsidiary management (separate admin per business unit)
    • Restricted management (sensitive groups in protected AU)
PowerShell - Administrative Units 
# Create Administrative Unit
New-MgDirectoryAdministrativeUnit -DisplayName "North America" `
    -Description "Users and groups in North America region"

# Add members to AU
$auId = (Get-MgDirectoryAdministrativeUnit -Filter "displayName eq 'North America'").Id
$userId = (Get-MgUser -Filter "userPrincipalName eq 'alice@contoso.com'").Id

New-MgDirectoryAdministrativeUnitMember -AdministrativeUnitId $auId `
    -DirectoryObjectId $userId

# Assign scoped role
$roleId = (Get-MgRoleManagementDirectoryRoleDefinition -Filter "displayName eq 'User Administrator'").Id

New-MgRoleManagementDirectoryRoleAssignment `
    -PrincipalId $adminUserId `
    -RoleDefinitionId $roleId `
    -DirectoryScopeId "/administrativeUnits/$auId"

Custom Domains

By default, tenants have a *.onmicrosoft.com domain. Custom domains allow users to sign in with your organization's domain name.

Adding a Custom Domain

  1. Add domain name in Entra admin center
  2. Create DNS verification record (TXT or MX)
  3. Verify domain ownership
  4. Configure DNS records for services (MX, CNAME, etc.)
  5. Optionally set as primary domain
DNS Records for Custom Domain 
# Verification TXT Record
Type: TXT
Host: @
Value: MS=ms12345678 (provided by Microsoft)
TTL: 3600

# Exchange Online (if using)
Type: MX
Host: @
Value: contoso-com.mail.protection.outlook.com
Priority: 0

# Autodiscover
Type: CNAME
Host: autodiscover
Value: autodiscover.outlook.com

# SPF Record (email authentication)
Type: TXT
Host: @
Value: v=spf1 include:spf.protection.outlook.com -all

Conditional Access Policies

Conditional Access is Microsoft's Zero Trust policy engine. It brings signals together to make access decisions, enforcing organizational policies at the point of access.

Conditional Access Architecture

Policy Components

    CONDITIONAL ACCESS POLICY STRUCTURE
    ═════════════════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                         POLICY                                       │
    │  Name: "Require MFA for All Users"                                  │
    │  State: Enabled / Disabled / Report-only                            │
    └─────────────────────────────────────────────────────────────────────┘
                                    │
         ┌──────────────────────────┴──────────────────────────┐
         │                                                      │
         ▼                                                      ▼
    ┌─────────────────────────────────┐    ┌─────────────────────────────────┐
    │       ASSIGNMENTS               │    │       ACCESS CONTROLS           │
    │       (IF this...)              │    │       (...THEN this)            │
    └─────────────────────────────────┘    └─────────────────────────────────┘
         │                                      │
         ├── Users and Groups                   ├── GRANT Controls
         │   • Include: All users               │   • Block access
         │   • Exclude: Break-glass accounts    │   • Grant access
         │   • Include: Specific groups         │     - Require MFA
         │   • Include: Directory roles         │     - Require compliant device
         │                                      │     - Require Hybrid Joined
         ├── Cloud Apps or Actions              │     - Require approved app
         │   • All cloud apps                   │     - Require app protection
         │   • Select specific apps             │     - Require password change
         │   • User actions (register info)     │     - Require ToU acceptance
         │   • Authentication context           │     - Require auth strength
         │                                      │   • Operator: AND / OR
         └── Conditions                         │
             • User risk (P2)                   └── SESSION Controls
             • Sign-in risk (P2)                    • App enforced restrictions
             • Device platforms                     • Conditional Access App Control
             • Locations                            • Sign-in frequency
             • Client apps                          • Persistent browser session
             • Filter for devices                   • Customize continuous access
             • Authentication flows                 • Disable resilience defaults

Assignment Details

Users and Groups

Define who the policy applies to.

Option Description
All users Every user in the directory (including guests)
Specific users/groups Named users or security groups
Directory roles Built-in roles (Global Admin, User Admin, etc.)
Guest or external users B2B guests, direct connect, local guest
Exclude Emergency access accounts, service accounts
🚨 Critical: Break-Glass Accounts

ALWAYS exclude at least 2 emergency access accounts from ALL Conditional Access policies. These accounts should be cloud-only, have long complex passwords, and not require MFA.

Cloud Apps or Actions

What the user is trying to access.

  • All cloud apps: Every app registered in Entra ID
  • Select apps: Specific enterprise or registered apps
  • User actions:
    • Register security information
    • Register or join devices
  • Authentication context: Custom contexts for step-up auth

Common App IDs:

  • Microsoft 365: Office365
  • Azure Management: 797f4846-ba00-4fd7-ba43-dac1f8f63013
  • Microsoft Graph: 00000003-0000-0000-c000-000000000000
  • Azure Portal: c44b4083-3bb0-49c1-b47d-974e53cbdf3c

Conditions

Additional context signals for policy evaluation.

Condition Options License
User risk High, Medium, Low, No risk P2
Sign-in risk High, Medium, Low, No risk P2
Device platforms Android, iOS, Windows, macOS, Linux P1
Locations Named locations (IP/Country) P1
Client apps Browser, Mobile apps, Desktop clients, Legacy P1
Device filter Rule expressions on device properties P1

Grant Controls

What is required for access to be granted.

  • Block access: Deny all access
  • Grant access with controls:
    • Require multi-factor authentication
    • Require authentication strength (specific methods)
    • Require device to be marked as compliant
    • Require Hybrid Azure AD joined device
    • Require approved client app
    • Require app protection policy
    • Require password change
    • Require terms of use

Control Operators:

  • Require all: All selected controls must be satisfied (AND)
  • Require one: Any one control satisfies the requirement (OR)

Session Controls

Controls applied during the session.

  • App enforced restrictions: Signals to apps (SharePoint/Exchange) for limited access
  • Conditional Access App Control: Route through Microsoft Defender for Cloud Apps
  • Sign-in frequency: How often users must re-authenticate (1-365 days)
  • Persistent browser session: Allow "stay signed in" or require session cookie only
  • Customize continuous access evaluation: Strictly enforce location policies
  • Disable resilience defaults: Don't use cached credentials during outages

Named Locations

Named locations define trusted IP ranges or countries for use in Conditional Access policies.

PowerShell - Create Named Location 
# Create IP-based named location (Corporate Network)
$params = @{
    "@odata.type" = "#microsoft.graph.ipNamedLocation"
    DisplayName = "Corporate Network"
    IsTrusted = $true
    IpRanges = @(
        @{
            "@odata.type" = "#microsoft.graph.iPv4CidrRange"
            CidrAddress = "203.0.113.0/24"
        }
        @{
            "@odata.type" = "#microsoft.graph.iPv4CidrRange"
            CidrAddress = "198.51.100.0/24"
        }
        @{
            "@odata.type" = "#microsoft.graph.iPv6CidrRange"
            CidrAddress = "2001:db8::/32"
        }
    )
}

New-MgIdentityConditionalAccessNamedLocation -BodyParameter $params

# Create country-based named location
$params = @{
    "@odata.type" = "#microsoft.graph.countryNamedLocation"
    DisplayName = "Allowed Countries"
    CountriesAndRegions = @("US", "CA", "GB", "DE")
    IncludeUnknownCountriesAndRegions = $false
}

New-MgIdentityConditionalAccessNamedLocation -BodyParameter $params

Common Policy Templates

Microsoft provides templates for common security scenarios. Here are production-ready policy configurations.

1. Require MFA for All Users +

Purpose: Foundational policy ensuring all users complete MFA.

Terraform 
resource "azuread_conditional_access_policy" "require_mfa_all_users" {
  display_name = "CA001: Require MFA for all users"
  state        = "enabledForReportingButNotEnforced"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
    }
    
    applications {
      include_applications = ["All"]
    }
    
    client_app_types = ["all"]
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["mfa"]
  }
}
2. Require Phishing-Resistant MFA for Admins +

Purpose: Higher security for privileged accounts using authentication strength.

Terraform 
resource "azuread_conditional_access_policy" "phishing_resistant_admins" {
  display_name = "CA002: Require phishing-resistant MFA for admins"
  state        = "enabled"

  conditions {
    users {
      include_roles = [
        "62e90394-69f5-4237-9190-012177145e10", # Global Administrator
        "194ae4cb-b126-40b2-bd5b-6091b380977d", # Security Administrator
        "f28a1f50-f6e7-4571-818b-6a12f2af6b6c", # SharePoint Administrator
        "29232cdf-9323-42fd-ade2-1d097af3e4de", # Exchange Administrator
        "9b895d92-2cd3-44c7-9d02-a6ac2d5ea5c3", # Application Administrator
      ]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
    }
    
    applications {
      include_applications = ["All"]
    }
  }

  grant_controls {
    operator                          = "OR"
    authentication_strength_policy_id = data.azuread_authentication_strength_policy.phishing_resistant.id
  }
}

data "azuread_authentication_strength_policy" "phishing_resistant" {
  display_name = "Phishing-resistant MFA"
}
3. Block Legacy Authentication +

Purpose: Block authentication protocols that don't support MFA (IMAP, SMTP, POP3, etc.).

Terraform 
resource "azuread_conditional_access_policy" "block_legacy_auth" {
  display_name = "CA003: Block legacy authentication"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
    }
    
    applications {
      include_applications = ["All"]
    }
    
    client_app_types = [
      "exchangeActiveSync",
      "other"
    ]
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["block"]
  }
}
4. Require Compliant Device +

Purpose: Require devices to meet Intune compliance policies.

Terraform 
resource "azuread_conditional_access_policy" "require_compliant_device" {
  display_name = "CA004: Require compliant device for sensitive apps"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
      exclude_groups = [
        azuread_group.ca_exclusion_compliant_device.object_id
      ]
    }
    
    applications {
      include_applications = [
        data.azuread_service_principal.sharepoint.object_id,
        data.azuread_service_principal.teams.object_id
      ]
    }
    
    platforms {
      include_platforms = ["windows", "macOS", "iOS", "android"]
    }
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["compliantDevice"]
  }
}
5. Block Access from Untrusted Locations +

Purpose: Block access from high-risk countries or unknown locations.

Terraform 
resource "azuread_conditional_access_policy" "block_untrusted_locations" {
  display_name = "CA005: Block access from blocked countries"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
    }
    
    applications {
      include_applications = ["All"]
    }
    
    locations {
      include_locations = [
        azuread_named_location.blocked_countries.id
      ]
    }
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["block"]
  }
}

resource "azuread_named_location" "blocked_countries" {
  display_name = "Blocked Countries"
  country {
    countries_and_regions = ["RU", "CN", "KP", "IR"]
    include_unknown_countries_and_regions = true
  }
}
6. Require Password Change for Risky Users +

Purpose: Force password reset when Identity Protection detects user risk.

Terraform 
resource "azuread_conditional_access_policy" "risky_user_password_change" {
  display_name = "CA006: Require password change for high-risk users"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = [
        azuread_user.break_glass_1.object_id,
        azuread_user.break_glass_2.object_id
      ]
    }
    
    applications {
      include_applications = ["All"]
    }
    
    user_risk_levels = ["high"]
  }

  grant_controls {
    operator          = "AND"
    built_in_controls = ["mfa", "passwordChange"]
  }
}

Report-Only Mode

Report-only mode allows testing policies without enforcing them. Results are logged in sign-in logs without affecting users.

    POLICY DEPLOYMENT WORKFLOW
    ═══════════════════════════

    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
    │   Create    │     │  Report-    │     │   Enable    │
    │   Policy    │────▶│  Only Mode  │────▶│   Policy    │
    │             │     │  (Test)     │     │  (Enforce)  │
    └─────────────┘     └──────┬──────┘     └─────────────┘
                               │
                               ▼
                    ┌─────────────────────┐
                    │  Monitor Sign-in    │
                    │  Logs for Impact    │
                    │                     │
                    │  • Would have been  │
                    │    applied: X users │
                    │  • Would have       │
                    │    blocked: Y       │
                    │  • Failures: Z      │
                    └─────────────────────┘

    SIGN-IN LOG REPORT-ONLY RESULTS:
    ─────────────────────────────────
    • reportOnlySuccess: User would have passed
    • reportOnlyFailure: User would have been blocked/required control
    • reportOnlyNotApplied: Policy wouldn't apply
    • reportOnlyInterrupted: User session interrupted before completion
KQL - Analyze Report-Only Impact 
// Analyze report-only policy impact
SigninLogs
| where TimeGenerated > ago(7d)
| mv-expand ConditionalAccessPolicies
| extend PolicyName = tostring(ConditionalAccessPolicies.displayName)
| extend PolicyResult = tostring(ConditionalAccessPolicies.result)
| where PolicyResult startswith "reportOnly"
| summarize 
    Count = count(),
    UniqueUsers = dcount(UserPrincipalName)
    by PolicyName, PolicyResult
| order by Count desc

// Users who would have been blocked
SigninLogs
| where TimeGenerated > ago(7d)
| mv-expand ConditionalAccessPolicies
| where ConditionalAccessPolicies.result == "reportOnlyFailure"
| extend PolicyName = tostring(ConditionalAccessPolicies.displayName)
| summarize BlockedAttempts = count() by UserPrincipalName, PolicyName
| order by BlockedAttempts desc

What If Tool

The What If tool simulates policy evaluation for specific scenarios without requiring actual sign-in attempts.

Using What If

  1. Navigate to Entra ID → Protection → Conditional Access → Policies
  2. Click "What If" in the toolbar
  3. Select user, application, and conditions to simulate
  4. Review which policies would apply and their results
✓ What If Best Practices
  • Test with representative users from each population
  • Simulate from different locations and device types
  • Include guest users in testing
  • Verify break-glass accounts are properly excluded

Conditional Access and Zero Trust

    ZERO TRUST PRINCIPLES WITH CONDITIONAL ACCESS
    ═══════════════════════════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    ZERO TRUST ARCHITECTURE                           │
    │                  "Never Trust, Always Verify"                        │
    └─────────────────────────────────────────────────────────────────────┘

    PRINCIPLE 1: VERIFY EXPLICITLY
    ────────────────────────────────
    • Authenticate and authorize based on ALL available data points
    • User identity → Conditional Access evaluates user/group
    • Device health → Require compliant/managed device
    • Location → Named locations, trusted IPs
    • Risk level → Identity Protection integration

    PRINCIPLE 2: USE LEAST PRIVILEGED ACCESS
    ─────────────────────────────────────────
    • Just-in-time access → PIM integration
    • Just-enough-access → Granular app assignments
    • Risk-based policies → Higher assurance for sensitive apps
    • Session controls → Limited session duration

    PRINCIPLE 3: ASSUME BREACH
    ──────────────────────────
    • Segment access → Different policies per app sensitivity
    • End-to-end encryption → Session controls
    • Continuous monitoring → Sign-in logs, risk detection
    • Anomaly detection → Identity Protection signals

    CONDITIONAL ACCESS IMPLEMENTATION:
    ──────────────────────────────────
    ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
    │    IDENTITY     │    │     DEVICE      │    │   APPLICATION   │
    │                 │    │                 │    │                 │
    │ • Strong auth   │    │ • Compliance    │    │ • App-specific  │
    │ • Risk-based    │    │ • Join status   │    │   policies      │
    │ • MFA required  │    │ • Health checks │    │ • Step-up auth  │
    └─────────────────┘    └─────────────────┘    └─────────────────┘
             │                      │                      │
             └──────────────────────┼──────────────────────┘
                                    │
                                    ▼
                        ┌─────────────────────┐
                        │  POLICY DECISION    │
                        │  (Allow/Block/       │
                        │   Require Controls) │
                        └─────────────────────┘

Infrastructure as Code

Complete Terraform module for deploying a baseline Conditional Access policy set.

Terraform - Complete CA Module 
# main.tf - Conditional Access Baseline Module

terraform {
  required_providers {
    azuread = {
      source  = "hashicorp/azuread"
      version = "~> 2.47"
    }
  }
}

# Variables
variable "break_glass_users" {
  description = "Object IDs of break-glass accounts to exclude from all policies"
  type        = list(string)
}

variable "trusted_locations_ips" {
  description = "List of trusted IP ranges in CIDR notation"
  type        = list(string)
  default     = []
}

variable "blocked_countries" {
  description = "List of country codes to block"
  type        = list(string)
  default     = ["RU", "CN", "KP", "IR"]
}

# Named Locations
resource "azuread_named_location" "corporate_network" {
  display_name = "Corporate Network - Trusted"
  ip {
    ip_ranges = var.trusted_locations_ips
    trusted   = true
  }
}

resource "azuread_named_location" "blocked_countries" {
  display_name = "Blocked Countries"
  country {
    countries_and_regions                 = var.blocked_countries
    include_unknown_countries_and_regions = true
  }
}

# CA Groups
resource "azuread_group" "ca_exclusion_group" {
  display_name     = "CA-Exclusion-MFA"
  security_enabled = true
  description      = "Users excluded from MFA policies (with approval)"
}

# Policy 1: Block Legacy Authentication
resource "azuread_conditional_access_policy" "block_legacy_auth" {
  display_name = "CA001: Block legacy authentication"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = var.break_glass_users
    }
    applications {
      include_applications = ["All"]
    }
    client_app_types = ["exchangeActiveSync", "other"]
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["block"]
  }
}

# Policy 2: Require MFA for All Users
resource "azuread_conditional_access_policy" "require_mfa_all" {
  display_name = "CA002: Require MFA for all users"
  state        = "enabled"

  conditions {
    users {
      include_users  = ["All"]
      exclude_users  = var.break_glass_users
      exclude_groups = [azuread_group.ca_exclusion_group.object_id]
    }
    applications {
      include_applications = ["All"]
    }
    client_app_types = ["browser", "mobileAppsAndDesktopClients"]
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["mfa"]
  }
}

# Policy 3: Block Untrusted Countries
resource "azuread_conditional_access_policy" "block_countries" {
  display_name = "CA003: Block access from blocked countries"
  state        = "enabled"

  conditions {
    users {
      include_users = ["All"]
      exclude_users = var.break_glass_users
    }
    applications {
      include_applications = ["All"]
    }
    locations {
      include_locations = [azuread_named_location.blocked_countries.id]
    }
  }

  grant_controls {
    operator          = "OR"
    built_in_controls = ["block"]
  }
}

# Outputs
output "policy_ids" {
  value = {
    block_legacy_auth = azuread_conditional_access_policy.block_legacy_auth.id
    require_mfa_all   = azuread_conditional_access_policy.require_mfa_all.id

}

Entra ID Authentication Methods

Microsoft Entra ID supports a wide range of authentication methods from traditional passwords to modern passwordless options. This chapter covers all available methods, their configuration, and best practices.

Method Type Phish-Resistant User Experience Deployment Effort

Passwordless Authentication

Passwordless authentication eliminates the primary attack vector in most breaches. Entra ID supports multiple passwordless methods.

FIDO2 Security Keys

Hardware security keys using the FIDO2/WebAuthn standard provide the strongest phishing-resistant authentication.

How FIDO2 Works

    FIDO2 AUTHENTICATION FLOW
    ══════════════════════════

    ┌────────┐          ┌────────────┐          ┌──────────┐
    │  User  │          │   Browser  │          │ Entra ID │
    │        │          │  /Device   │          │          │
    └───┬────┘          └─────┬──────┘          └────┬─────┘
        │                     │                      │
        │ 1. Navigate to      │                      │
        │    sign-in page     │                      │
        │────────────────────▶│                      │
        │                     │ 2. Request challenge│
        │                     │─────────────────────▶│
        │                     │                      │
        │                     │ 3. Challenge         │
        │                     │◀─────────────────────│
        │                     │                      │
        │ 4. Touch security   │                      │
        │    key              │                      │
        │────────────────────▶│                      │
        │                     │                      │
        │    ┌────────────────┴────────────────┐    │
        │    │ Security Key:                    │    │
        │    │ • Verifies origin (domain)      │    │
        │    │ • Signs challenge with          │    │
        │    │   private key                   │    │
        │    │ • Private key NEVER leaves key  │    │
        │    └────────────────┬────────────────┘    │
        │                     │                      │
        │                     │ 5. Signed assertion │
        │                     │─────────────────────▶│
        │                     │                      │
        │                     │    6. Verify with   │
        │                     │       public key    │
        │                     │                      │
        │                     │ 7. Access granted   │
        │◀────────────────────│◀─────────────────────│
        │                     │                      │

    WHY IT'S PHISHING-RESISTANT:
    • Key is bound to specific domain (origin)
    • Fake sites get different origin → key won't respond
    • Private key never transmitted or exposed

Supported FIDO2 Keys

  • YubiKey 5 Series: USB-A, USB-C, NFC options
  • Feitian: Various form factors
  • AuthenTrend: Biometric options
  • Thales: Enterprise-grade keys
PowerShell - Enable FIDO2 
# Enable FIDO2 security keys for all users
$params = @{
    "@odata.type" = "#microsoft.graph.fido2AuthenticationMethodConfiguration"
    State = "enabled"
    IsAttestationEnforced = $true
    IsSelfServiceRegistrationAllowed = $true
}

Update-MgPolicyAuthenticationMethodPolicyAuthenticationMethodConfiguration `
    -AuthenticationMethodConfigurationId "Fido2" `
    -BodyParameter $params

Windows Hello for Business

Device-bound credential using biometrics (face/fingerprint) or PIN. Provides passwordless authentication for Windows devices.

Deployment Models

  • Cloud-only: Entra ID Joined devices, no on-prem requirements
  • Hybrid Key Trust: Requires DC update, simpler than cert trust
  • Hybrid Certificate Trust: Requires PKI, supports RDP
  • On-premises Key Trust: AD-only environments
✓ WHfB Security Features
  • Credential is device-bound (TPM protected)
  • Biometric data never leaves device
  • PIN is device-specific, not a network credential
  • Anti-hammering protections
Intune Policy - WHfB 
// Intune Configuration Profile - Windows Hello for Business
{
    "windowsHelloForBusinessEnabled": true,
    "pinMinimumLength": 6,
    "pinMaximumLength": 127,
    "pinUppercaseCharactersUsage": "allowed",
    "pinLowercaseCharactersUsage": "allowed", 
    "pinSpecialCharactersUsage": "allowed",
    "securityDeviceRequired": true,  // Require TPM
    "unlockWithBiometricsEnabled": true,
    "enhancedAntiSpoofingForFacialFeaturesEnabled": true
}

Microsoft Authenticator Passwordless

Phone sign-in allows users to sign in without a password using the Authenticator app.

Requirements

  • Microsoft Authenticator app (latest version)
  • iOS 8.0+ or Android 6.0+
  • Device registered with Entra ID
  • Account added to Authenticator app

Number Matching (Required)

Number matching requires users to enter a number displayed on the sign-in screen into the Authenticator app, preventing accidental approvals and MFA fatigue attacks.

PowerShell - Configure Authenticator 
# Enable Microsoft Authenticator with passwordless
$params = @{
    "@odata.type" = "#microsoft.graph.microsoftAuthenticatorAuthenticationMethodConfiguration"
    State = "enabled"
    FeatureSettings = @{
        DisplayAppInformationRequiredState = @{
            State = "enabled"  # Show app name during approval
        }
        DisplayLocationInformationRequiredState = @{
            State = "enabled"  # Show location during approval
        }
        NumberMatchingRequiredState = @{
            State = "enabled"  # Require number matching
        }
    }
    IncludeTargets = @(
        @{
            TargetType = "group"
            Id = "all_users"
            AuthenticationMode = "any"  # or "deviceBasedPush" for passwordless
        }
    )
}

Update-MgPolicyAuthenticationMethodPolicyAuthenticationMethodConfiguration `
    -AuthenticationMethodConfigurationId "MicrosoftAuthenticator" `
    -BodyParameter $params

Certificate-Based Authentication (CBA)

Authenticate using X.509 certificates stored on smart cards or devices. Requires PKI infrastructure.

Configuration Steps

  1. Upload trusted CA certificates to Entra ID
  2. Configure certificate binding (UPN or certificate mapping)
  3. Enable CBA authentication method
  4. Configure authentication strength policies
  5. Deploy certificates to users (smart cards or device certs)
💡 CBA Use Cases
  • Government/regulated industries (PIV/CAC cards)
  • High-security environments
  • Shared workstation scenarios
  • Organizations with existing PKI investment

Multi-Factor Authentication (MFA)

MFA requires two or more verification factors. Entra ID supports various MFA methods with different security levels.

MFA Method Comparison

Method Security User Experience Risks
Authenticator Push High Best (one tap) MFA fatigue without number matching
TOTP (6-digit code) High Good Phishable (real-time proxy)
OATH Hardware Token High Fair Token management overhead
SMS Low Good SIM swap, SS7 vulnerabilities
Voice Call Low Fair Call forwarding, social engineering

Per-User MFA vs Conditional Access MFA

    MFA ENFORCEMENT METHODS
    ════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │  PER-USER MFA (Legacy - Not Recommended)                            │
    │  ───────────────────────────────────────────────────────────────    │
    │  • Enabled per individual user                                      │
    │  • All-or-nothing (MFA always required)                            │
    │  • No context awareness (location, device, app)                     │
    │  • Managed in legacy portal                                         │
    │  • Being deprecated                                                 │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │  CONDITIONAL ACCESS MFA (Recommended)                               │
    │  ───────────────────────────────────────────────────────────────    │
    │  • Policy-based enforcement                                         │
    │  • Context-aware (location, device state, risk, app)               │
    │  • Granular control                                                │
    │  • Can be combined with other controls                              │
    │  • Requires P1 license                                              │
    │                                                                     │
    │  Example Policies:                                                  │
    │  • Require MFA for all users accessing any cloud app               │
    │  • Require MFA only from untrusted locations                       │
    │  • Require MFA only for sensitive applications                     │
    │  • Skip MFA on compliant devices from trusted locations            │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │  SECURITY DEFAULTS (Free Tier)                                      │
    │  ───────────────────────────────────────────────────────────────    │
    │  • Basic MFA for all users                                         │
    │  • Block legacy authentication                                      │
    │  • Require MFA for admins                                          │
    │  • Cannot customize                                                 │
    │  • Good starting point for small organizations                     │
    └─────────────────────────────────────────────────────────────────────┘
PowerShell - MFA Registration Policy 
# Get current authentication methods policy
Get-MgPolicyAuthenticationMethodPolicy

# Configure MFA registration campaign
$params = @{
    RegistrationEnforcement = @{
        AuthenticationMethodsRegistrationCampaign = @{
            SnoozeDurationInDays = 1
            State = "enabled"
            ExcludeTargets = @()
            IncludeTargets = @(
                @{
                    Id = "all_users"
                    TargetType = "group"
                    TargetedAuthenticationMethod = "microsoftAuthenticator"
                }
            )
        }
    }
}

Update-MgPolicyAuthenticationMethodPolicy -BodyParameter $params

⚠️ SSPR Security Considerations
  • Security questions alone are NOT recommended (easily researched)
  • Require at least 2 authentication methods
  • Consider blocking SSPR for privileged accounts
  • Monitor SSPR events for abuse
  • Enable password writeback for hybrid environments

PowerShell - Custom Banned Passwords 
# View current password policy settings
Get-MgPolicyAuthenticationMethodPolicy

# Configure custom banned password list
$params = @{
    EnableBannedPasswordCheck = $true
    BannedPasswordCheckOnPremisesMode = "Enforce"  # or "Audit"
    CustomBannedPasswords = @(
        "contoso"
        "contoso123"
        "companyname"
        "winter2024"
        "summer2024"
        "project"
        "password"
    )
}

Authentication Strengths

Authentication Strengths allow you to define which authentication method combinations satisfy MFA requirements in Conditional Access policies.

Built-in Authentication Strengths

Strength Allowed Methods Use Case
MFA Strength Any MFA method (push, TOTP, SMS, voice, etc.) General MFA requirement
Passwordless MFA FIDO2, WHfB, Authenticator passwordless, Certificate Higher security, no password
Phishing-Resistant MFA FIDO2, WHfB, Certificate only Highest security (admins, sensitive apps)
JSON - Custom Authentication Strength 
// Create custom authentication strength via Graph API
POST https://graph.microsoft.com/v1.0/policies/authenticationStrengthPolicies

{
    "displayName": "Executives - Phishing Resistant",
    "description": "Requires phishing-resistant authentication for executive accounts",
    "allowedCombinations": [
        "fido2",
        "windowsHelloForBusiness",
        "x509CertificateMultiFactor"
    ],
    "requirementsSatisfied": "mfa"
}

Troubleshooting Authentication

Common Issues and Solutions

User can't register for MFA +

Check:

  • Authentication methods policy includes user's group
  • User has appropriate license
  • Combined registration is enabled
  • User is not blocked by Conditional Access

Diagnostic:

Get-MgUser -UserId "user@domain.com" -Property AuthenticationMethods
MFA prompt every time (no SSO) +

Check:

  • Conditional Access sign-in frequency settings
  • Persistent browser session settings
  • Application session timeout configuration
  • Token lifetime policies
FIDO2 key not working +

Check:

  • Key AAGUID is allowed in policy
  • Browser supports WebAuthn (Edge, Chrome, Firefox)
  • User has registered key in Security Info
  • Key firmware is up to date

Sign-in Log Analysis

KQL - Authentication Analysis 
// Failed sign-ins by authentication method
SigninLogs
| where TimeGenerated > ago(7d)
| where ResultType != 0
| summarize FailureCount = count() by 
    AuthenticationMethodsUsed, 
    ResultDescription
| order by FailureCount desc

// MFA challenge outcomes
SigninLogs
| where TimeGenerated > ago(7d)
| mv-expand AuthenticationDetails
| extend AuthMethod = tostring(AuthenticationDetails.authenticationMethod)
| extend AuthResult = tostring(AuthenticationDetails.succeeded)
| summarize count() by AuthMethod, AuthResult

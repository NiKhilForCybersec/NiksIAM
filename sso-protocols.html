SSO Protocols Deep Dive

This chapter provides in-depth coverage of SAML 2.0, OAuth 2.0, and OpenID Connect protocols used for single sign-on and API authorization.

On This Page
  • SAML 2.0
  • OAuth 2.0
  • OpenID Connect

SAML 2.0

    SAML 2.0 SP-INITIATED FLOW
    ═══════════════════════════

    ┌────────┐          ┌─────────────┐          ┌────────────┐
    │  User  │          │   Service   │          │  Identity  │
    │Browser │          │  Provider   │          │  Provider  │
    └───┬────┘          └──────┬──────┘          └─────┬──────┘
        │                      │                       │
        │ 1. Access protected  │                       │
        │    resource          │                       │
        │─────────────────────▶│                       │
        │                      │                       │
        │ 2. Redirect with     │                       │
        │    SAML AuthnRequest │                       │
        │◀─────────────────────│                       │
        │                      │                       │
        │ 3. AuthnRequest (via browser redirect)      │
        │─────────────────────────────────────────────▶│
        │                      │                       │
        │                      │    4. User            │
        │                      │    authenticates      │
        │◀─────────────────────────────────────────────│
        │                      │                       │
        │ 5. SAML Response with signed Assertion      │
        │◀─────────────────────────────────────────────│
        │                      │                       │
        │ 6. POST Assertion    │                       │
        │    to ACS URL        │                       │
        │─────────────────────▶│                       │
        │                      │                       │
        │                      │ 7. Validate signature │
        │                      │    Create session     │
        │                      │                       │
        │ 8. Access granted    │                       │
        │◀─────────────────────│                       │
        │                      │                       │

    KEY COMPONENTS:
    ═══════════════
    • AuthnRequest: SP asks IdP to authenticate user
    • SAML Response: IdP's answer containing assertion
    • Assertion: Signed XML with user identity + attributes
    • ACS URL: Assertion Consumer Service endpoint at SP

OpenID Connect

OIDC adds identity layer on top of OAuth 2.0, providing user authentication.

JWT ID Token Structure 
// ID Token (JWT) Example
{
  // Header
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-id-123"
}
.
{
  // Payload (Claims)
  "iss": "https://idp.example.com",
  "sub": "user-123",
  "aud": "client-app-id",
  "exp": 1699999999,
  "iat": 1699996399,
  "nonce": "random-nonce",
  "email": "user@example.com",
  "name": "John Doe"
}
.

Protocol Comparison

Aspect SAML 2.0 OAuth 2.0 OIDC
Purpose Authentication + Authorization Authorization only Authentication + Authorization
Token Format XML Opaque or JWT JWT (ID Token)
Best For Enterprise web SSO API authorization Modern apps + APIs
Mobile Support Limited Excellent Excellent

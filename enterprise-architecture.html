Security Use Cases - Detailed Flows

Understanding how authentication, authorization, and security controls work together. Each flow shows the complete journey of a request through the security stack, including protocols, ports, and decision points.

🌐 Scenario: Employee Accessing Salesforce from Home

A remote employee on their corporate laptop wants to access Salesforce CRM. The request must pass through identity verification, policy evaluation, and continuous monitoring.

STEP 1 User Opens Application

User clicks Salesforce bookmark or navigates to https://company.my.salesforce.com

Protocol: HTTPS :443 Component: Browser
STEP 2 SP Redirects to IdP

Salesforce (Service Provider) detects no valid session and redirects browser to Entra ID (Identity Provider) with a SAML AuthnRequest.

Protocol: SAML 2.0 Redirect: HTTP 302 To: login.microsoftonline.com :443
STEP 3 Primary Authentication

Entra ID presents login page. User enters UPN (user@company.com). Password verified against Password Hash Sync (PHS) or Pass-Through Auth (PTA) if hybrid.

Method: Password or Passwordless Component: Entra ID Hybrid: PHS/PTA :443
STEP 4 MFA Challenge

MFA is required by Conditional Access policy. User receives push notification to Microsoft Authenticator app with number matching challenge.

Method: Push + Number Match Alternative: FIDO2, WHfB, TOTP Timeout: 60 seconds
STEP 5 Conditional Access Evaluation

CA engine evaluates all signals in real-time to make access decision:

User Risk
Low (Identity Protection)
Sign-in Risk
None detected
Device
Intune Compliant ✓
Location
Canada (allowed)
Application
Salesforce (approved)
Client App
Browser (modern auth)
Decision: GRANT ACCESS - All conditions met
STEP 6 Token Issuance

Entra ID generates SAML Response containing signed assertion with user claims (UPN, groups, roles) and redirects back to Salesforce.

Token: SAML 2.0 Assertion Lifetime: 1 hour Claims: UPN, Groups, Email
STEP 7 Application Access Granted

Salesforce validates SAML signature, creates session, maps user to Salesforce profile based on group claims, grants access to CRM data.

Session: Salesforce session cookie Role: Sales User
STEP 8 Continuous Monitoring

Throughout the session, multiple security controls monitor activity:

DLP (Purview)
Monitors downloads, blocks PII export
CASB (MCAS)
Session controls, anomaly detection
SIEM (Sentinel)
Sign-in logs, audit trail
📊 Complete Request Flow Summary
User Browser → Salesforce (HTTPS :443) → 302 Redirect → Entra ID (HTTPS :443)
→ Password Auth → MFA Push → Conditional Access Eval → SAML Token
→ Salesforce ACS URL → Session Created → DLP/CASB Monitor → SIEM Log

🏢 Scenario: Remote User Accessing On-Premises Web Application

A remote employee needs to access a legacy internal HR application hosted on-premises. The connection goes through Entra App Proxy without requiring VPN.

STEP 1 User Accesses External URL

User navigates to https://hrportal-company.msappproxy.net (published App Proxy URL).

Protocol: HTTPS :443 Service: Azure App Proxy
STEP 2 Pre-Authentication at Cloud Edge

App Proxy service in Azure intercepts request, requires Entra ID authentication BEFORE reaching internal network. This is "pre-authentication" - blocking unauthenticated traffic.

Pre-Auth: Entra ID No VPN Required MFA Enforced
STEP 3 Conditional Access Evaluation

CA policy for on-prem apps via App Proxy may have stricter requirements:

Require
Hybrid Entra Joined Device
Require
Compliant Device
MFA
Phishing-resistant
STEP 4 Cloud-to-Connector Communication

Azure App Proxy cloud service sends request to App Proxy Connector installed in your DMZ. The connector makes OUTBOUND-ONLY connections - no inbound firewall rules needed!

Connector: OUTBOUND :443 only No Inbound Ports Service Bus Relay
STEP 5 Connector to Internal Application

Connector forwards request to internal HR portal (http://hrportal.corp.local:80). Can use Kerberos Constrained Delegation (KCD) for SSO to Windows Integrated Auth apps.

Internal: HTTP/HTTPS SSO: KCD or Header-based No Changes to App
STEP 6 Logging & Monitoring

All access is logged in Entra ID sign-in logs and can be forwarded to SIEM:

Entra Sign-in Logs App Proxy Audit Logs Sentinel Integration

🔐 Scenario: IT Admin Accessing Production Server via PAM

A system administrator needs to perform emergency maintenance on a production database server. Access requires Just-In-Time (JIT) privilege elevation through PIM, manager approval, and full session recording.

STEP 1 Admin Requests Elevated Access

Admin logs into Entra portal, navigates to PIM, and requests activation of "Server Administrators" role. Must provide business justification and ticket number.

Portal: portal.azure.com Service: Entra PIM Justification Required
STEP 2 Phishing-Resistant MFA Required

PIM requires Authentication Strength policy: phishing-resistant MFA only. Admin must use FIDO2 security key or Windows Hello for Business - push notifications not accepted for privileged access.

FIDO2 Key: YubiKey 5 Protocol: WebAuthn :443 Auth Strength: Phishing-resistant
STEP 3 Approval Workflow Triggered

PIM sends approval request to designated approvers (IT Manager, Security Team). Email and Teams notifications sent. Approvers have 4 hours to respond.

Approver 1
IT Manager (Primary)
Approver 2
Security Team (Backup)
Timeout
4 hours max
STEP 4 Manager Approves Request

IT Manager reviews justification, verifies ticket in ServiceNow, and approves the request. Approval is logged with timestamp and approver identity.

APPROVED - Role activated for 8 hours maximum
STEP 5 JIT Role Activation

Admin is added to "Server Administrators" Azure AD group. This grants access to PAM portal (CyberArk) where server credentials are vaulted.

Duration: 8 hours Auto-Expire: Yes Audit: Full Trail
STEP 6 PAM Vault Credential Retrieval

Admin accesses CyberArk PVWA portal. Rather than viewing the password, admin initiates a PSM (Privileged Session Manager) connection. CyberArk injects credentials automatically.

PAM: CyberArk Port: HTTPS :443 Vault: :1858 Password: Never Shown
STEP 7 PSM Brokered Session

CyberArk PSM establishes RDP session to production server on admin's behalf. Admin never directly connects; PSM acts as jump server with full session recording.

Protocol: RDP :3389 Via: PSM Server Recording: Video + Keystrokes
STEP 8 Full Session Recording

Every action is recorded: video of screen, all keystrokes, commands executed, files accessed. Recording stored for 7 years for compliance. Searchable by command.

Video
Full Screen
Keystrokes
All Input
Commands
Searchable
Retention
7 Years
STEP 9 Automatic Expiration & Password Rotation

After 8 hours (or when admin disconnects), access automatically revoked. CyberArk rotates the server password immediately, ensuring one-time use.

Role: Auto-Deactivated Password: Rotated Audit: Complete

🔒 Scenario: Remote User Connecting via VPN

A remote employee needs full network access to internal resources. This shows the complete VPN connection flow with RADIUS authentication, device posture checks, and network segmentation.

STEP 1 User Launches VPN Client

User opens GlobalProtect (or AnyConnect/FortiClient) VPN client, selects corporate gateway, and initiates connection.

Client: GlobalProtect Gateway: vpn.company.com
STEP 2 Device Posture Assessment

Before authentication, VPN gateway (Palo Alto) checks device compliance via HIP (Host Information Profile):

Antivirus
MDE Active ✓
Disk Encryption
BitLocker On ✓
Firewall
Enabled ✓
OS Patch
Current ✓
STEP 3 SAML Authentication to IdP

VPN gateway redirects to Entra ID for SAML authentication. User enters credentials in embedded browser window.

Protocol: SAML 2.0 IdP: Entra ID Port: HTTPS :443
STEP 4 MFA Challenge

Conditional Access requires MFA for VPN connections. User completes push notification with number matching.

Method: Push + Number Match CA Policy: Require MFA
STEP 5 Group-Based Network Assignment

VPN gateway receives SAML assertion with group claims. Based on user's AD groups, assigns appropriate network segment and firewall policies:

Sales Team
→ VLAN 100, CRM access only
Engineering
→ VLAN 200, Dev servers
IT Admins
→ VLAN 10, Full access
STEP 6 IPSec/SSL Tunnel Established

Encrypted tunnel established between client and VPN gateway. Split tunneling may route only corporate traffic through tunnel.

Protocol: IPSec or SSL Ports: UDP :4501, :500 Encryption: AES-256-GCM
STEP 7 Continuous Monitoring

VPN session monitored throughout connection:

Firewall: Traffic inspection SIEM: Connection logs EDR: Endpoint monitoring Timeout: 8hr max / 30min idle

⚙️ Scenario: Azure Function Accessing Microsoft Graph API

A serverless Azure Function needs to read user data from Microsoft Graph. Using Managed Identity eliminates the need for stored credentials - no secrets in code!

STEP 1 Function Triggers Execution

Azure Function triggered by HTTP request or timer. Code needs to call Microsoft Graph API to read user profiles.

Trigger: HTTP/Timer Runtime: .NET 8
STEP 2 Request Token via Managed Identity

Function code calls the Instance Metadata Service (IMDS) to request an access token. No credentials needed - Azure handles authentication automatically.

// C# Example - Managed Identity
var credential = new DefaultAzureCredential();
var token = await credential.GetTokenAsync(
  new TokenRequestContext(new[] { "https://graph.microsoft.com/.default" })
);
IMDS: 169.254.169.254:80 No Secrets in Code
STEP 3 Entra ID Issues OAuth Token

Azure fabric authenticates the Managed Identity to Entra ID. Entra ID issues an OAuth 2.0 access token with the requested scopes.

Grant: Client Credentials Token Type: Bearer (JWT) Lifetime: 1 hour
STEP 4 Token Contains Permission Scopes

JWT token includes the Graph API permissions granted to the Managed Identity (via App Registration in Entra ID):

Scope
User.Read.All
Permission Type
Application
Admin Consent
Granted ✓
STEP 5 Function Calls Graph API

Function makes HTTP request to Graph API with Bearer token in Authorization header:

GET https://graph.microsoft.com/v1.0/users
Authorization: Bearer eyJ0eXAiOiJKV1Qi...
API: graph.microsoft.com :443 Method: GET/POST
STEP 6 Graph API Validates & Responds

Graph API validates token signature, checks expiry, verifies scopes, and returns requested data.

200 OK - User data returned as JSON
STEP 7 Audit Logging

All service principal authentication and API calls logged in Entra ID and can be forwarded to SIEM:

Service Principal Sign-ins Graph API Audit Logs Function App Logs

🚨 Scenario: Automated Response to Compromised Account

EDR detects malware on an endpoint. SIEM correlates with suspicious sign-in. SOAR automatically contains the threat by isolating the device, revoking sessions, and creating incident ticket.

🔍 DETECTION PHASE
STEP 1 EDR Detects Malicious Activity

Microsoft Defender for Endpoint detects Mimikatz execution on workstation WIN-PC-1042. Alert severity: HIGH. MITRE ATT&CK: T1003 (Credential Dumping).

Source: MDE Severity: High MITRE: T1003
STEP 2 Alert Forwarded to SIEM

MDE alert streams to Microsoft Sentinel via native connector. Creates incident in Sentinel workspace.

SIEM: Sentinel Connector: M365 Defender Incident Created
STEP 3 SIEM Correlation & Enrichment

Sentinel correlates with other signals and enriches the incident:

Entra Sign-in Logs
Suspicious login from Russia 2hr ago
Identity Protection
User risk: HIGH
UEBA
Anomalous behavior score: 87
⚡ RESPONSE PHASE (AUTOMATED)
STEP 4 SOAR Playbook Triggers

Sentinel Logic App playbook "Respond-CredentialTheft" automatically triggers based on incident criteria: High severity + T1003 technique.

Playbook: Logic App Trigger: Automatic No Human Delay
STEP 5 Device Isolation

Playbook calls MDE API to isolate device from network. Device can only communicate with MDE service for remediation.

POST https://api.securitycenter.microsoft.com/api/machines/{machineId}/isolate
{ "Comment": "Automated isolation - credential theft detected", "IsolationType": "Full" }
STEP 6 Revoke All User Sessions

Playbook calls Microsoft Graph API to revoke all refresh tokens, forcing re-authentication everywhere.

POST https://graph.microsoft.com/v1.0/users/{userId}/revokeSignInSessions
STEP 7 Force Password Reset

Playbook forces password reset on next sign-in via Graph API.

forceChangePasswordNextSignIn: true
STEP 8 Create ServiceNow Ticket

Playbook creates Security Incident ticket in ServiceNow with all context, assigns to SOC team.

Ticket: SEC0012345 Priority: P1 Assigned: SOC Team
STEP 9 Notify Security Team

Playbook posts to SOC Teams channel with incident summary, actions taken, and next steps.

CONTAINMENT COMPLETE
Time from detection to containment: ~45 seconds (automated)

An employee attempts to upload a spreadsheet containing customer PII to their personal Dropbox. DLP detects sensitive content, blocks the upload, and alerts security team.

STEP 1 User Attempts Upload

Employee on corporate laptop opens browser, navigates to dropbox.com, and attempts to upload "Q4-Customer-List.xlsx".

Device: Corporate Managed Destination: dropbox.com
STEP 2 Endpoint DLP Intercepts

Microsoft Purview Endpoint DLP agent on the device intercepts the upload before it leaves the endpoint.

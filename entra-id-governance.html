Identity Governance

Microsoft Entra ID Governance provides capabilities to manage the identity lifecycle, govern access, and ensure compliance. It includes Privileged Identity Management, Access Reviews, Entitlement Management, and Lifecycle Workflows.

  • Access Reviews

Privileged Identity Management (PIM)

PIM provides just-in-time privileged access to reduce the risk of excessive permissions. Users request activation of privileged roles only when needed.

    PRIVILEGED IDENTITY MANAGEMENT
    ═══════════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                    PIM ROLE ASSIGNMENT TYPES                         │
    │                                                                      │
    │   ELIGIBLE                              ACTIVE                       │
    │   ────────                              ──────                       │
    │   • User CAN activate role              • User HAS role now         │
    │   • Must request activation             • No activation needed      │
    │   • Time-limited activation             • Can be time-bound         │
    │   • May require approval                • Immediate access          │
    │   • May require justification           • Use for service accounts  │
    │   • Preferred for humans                • Audit regularly           │
    │                                                                      │
    └─────────────────────────────────────────────────────────────────────┘

    PIM ACTIVATION FLOW:
    ═════════════════════

    ┌────────┐     ┌─────────────┐     ┌─────────────┐     ┌────────────┐
    │  User  │────▶│  Request    │────▶│  Approval   │────▶│   Active   │
    │(Eligible)    │  Activation │     │  (if req'd) │     │    Role    │
    └────────┘     └─────────────┘     └─────────────┘     └────────────┘
                         │                   │                    │
                         ▼                   ▼                    ▼
                   ┌───────────┐       ┌───────────┐       ┌───────────┐
                   │ Provide   │       │ Approver  │       │ Role      │
                   │ Justifi-  │       │ Reviews & │       │ Auto-     │
                   │ cation    │       │ Approves  │       │ Expires   │
                   └───────────┘       └───────────┘       └───────────┘

    SUPPORTED SCOPES:
    ─────────────────
    • Entra ID Roles (Global Admin, User Admin, etc.)
    • Azure Resource Roles (Subscription, Resource Group, Resource)
    • PIM for Groups (Security groups, M365 groups)

PIM Role Settings

Setting Description Recommended
Activation maximum duration How long role stays active after activation 8 hours max
Require MFA on activation Force MFA when activating role Yes (always)
Require justification User must provide reason for activation Yes
Require approval Designated approvers must approve activation Yes for high-privilege
Assignment expiration Eligible assignments expire after period 6-12 months
PowerShell - PIM Role Assignment 
# Get Entra ID role definition
$roleDefinition = Get-MgRoleManagementDirectoryRoleDefinition -Filter "displayName eq 'Global Administrator'"

# Create eligible PIM assignment
$params = @{
    Action = "adminAssign"
    PrincipalId = "user-object-id"
    RoleDefinitionId = $roleDefinition.Id
    DirectoryScopeId = "/"
    ScheduleInfo = @{
        StartDateTime = (Get-Date).ToUniversalTime().ToString("yyyy-MM-ddTHH:mm:ssZ")
        Expiration = @{
            Type = "AfterDuration"
            Duration = "P180D"  # 180 days
        }
    }
    Justification = "Annual role assignment renewal"
}

New-MgRoleManagementDirectoryRoleEligibilityScheduleRequest -BodyParameter $params

# Activate eligible role (as user)
$activationParams = @{
    Action = "selfActivate"
    PrincipalId = "my-object-id"
    RoleDefinitionId = $roleDefinition.Id
    DirectoryScopeId = "/"
    ScheduleInfo = @{
        StartDateTime = (Get-Date).ToUniversalTime().ToString("yyyy-MM-ddTHH:mm:ssZ")
        Expiration = @{
            Type = "AfterDuration"
            Duration = "PT8H"  # 8 hours
        }
    }
    Justification = "Need to update security policies"
}

New-MgRoleManagementDirectoryRoleAssignmentScheduleRequest -BodyParameter $activationParams

Access Reviews

Access Reviews ensure users have appropriate access by requiring periodic review and recertification of group memberships, app assignments, and role assignments.

    ACCESS REVIEW TYPES
    ════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │  GROUP MEMBERSHIP REVIEW                                             │
    │  ────────────────────────                                            │
    │  Who: Members of security/M365 groups                               │
    │  Reviewers: Group owners, specific users, self-review               │
    │  Scope: All members or guest users only                             │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │  APPLICATION ACCESS REVIEW                                           │
    │  ──────────────────────────                                          │
    │  Who: Users assigned to enterprise applications                     │
    │  Reviewers: App owners, managers, specific users                    │
    │  Scope: All assigned users or specific groups                       │
    └─────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────┐
    │  ROLE ASSIGNMENT REVIEW (PIM)                                        │
    │  ─────────────────────────────                                       │
    │  Who: Users with Entra ID or Azure resource roles                   │
    │  Reviewers: Role admins, specific users, self-review                │
    │  Scope: Active and/or eligible assignments                          │
    └─────────────────────────────────────────────────────────────────────┘

    REVIEW WORKFLOW:
    ════════════════

    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │  Review  │───▶│ Reviewers│───▶│ Approve/ │───▶│  Apply   │
    │  Created │    │ Notified │    │  Deny    │    │ Changes  │
    └──────────┘    └──────────┘    └──────────┘    └──────────┘
         │               │               │               │
         │               │               │               │
         ▼               ▼               ▼               ▼
    Recurrence     Email with      Justification    Auto-remove
    (quarterly)    review link     required         denied access
PowerShell - Create Access Review 
# Create quarterly group membership review
$params = @{
    DisplayName = "Quarterly Security Group Review"
    DescriptionForAdmins = "Review security group memberships"
    DescriptionForReviewers = "Please review if these users still need access"
    Scope = @{
        "@odata.type" = "#microsoft.graph.accessReviewQueryScope"
        Query = "/groups/group-id/members"
        QueryType = "MicrosoftGraph"
    }
    Reviewers = @(
        @{
            Query = "/groups/group-id/owners"
            QueryType = "MicrosoftGraph"
        }
    )
    Settings = @{
        MailNotificationsEnabled = $true
        ReminderNotificationsEnabled = $true
        JustificationRequiredOnApproval = $true
        DefaultDecisionEnabled = $true
        DefaultDecision = "Deny"
        InstanceDurationInDays = 14
        AutoApplyDecisionsEnabled = $true
        RecommendationsEnabled = $true
        Recurrence = @{
            Pattern = @{
                Type = "absoluteMonthly"
                Interval = 3
                DayOfMonth = 1
            }
            Range = @{
                Type = "noEnd"
                StartDate = (Get-Date).ToString("yyyy-MM-dd")
            }
        }
    }
}

New-MgIdentityGovernanceAccessReviewDefinition -BodyParameter $params

Entitlement Management

Entitlement Management automates access request workflows using access packages that bundle resources together.

    ENTITLEMENT MANAGEMENT STRUCTURE
    ═════════════════════════════════

    ┌─────────────────────────────────────────────────────────────────────┐
    │                          CATALOG                                     │
    │                    (Container for resources)                         │
    │                                                                      │
    │   ┌───────────────────────────────────────────────────────────────┐ │
    │   │                    ACCESS PACKAGE                              │ │
    │   │               "Finance Team Access"                            │ │
    │   │                                                                │ │
    │   │   RESOURCES:                    POLICIES:                      │ │
    │   │   ┌─────────────┐              ┌──────────────────────────┐   │ │
    │   │   │ Finance     │              │ • Who can request        │   │ │
    │   │   │ SharePoint  │              │ • Approval workflow      │   │ │
    │   │   │ Site        │              │ • Assignment duration    │   │ │
    │   │   ├─────────────┤              │ • Access review settings │   │ │
    │   │   │ Finance     │              └──────────────────────────┘   │ │
    │   │   │ App         │                                              │ │
    │   │   ├─────────────┤              REQUESTORS:                     │ │
    │   │   │ Finance     │              ┌──────────────────────────┐   │ │

Lifecycle Workflows

Lifecycle Workflows automate joiner, mover, and leaver processes for employee lifecycle management.

Workflow Type Trigger Common Tasks
Joiner X days before/after employee start date Generate TAP, send welcome email, add to groups
Mover Job change, department change Update groups, trigger access review
Leaver X days before/after employee end date Remove groups, disable account, revoke access
JSON - Lifecycle Workflow Example 
// Leaver workflow - 7 days before last day
{
    "displayName": "Pre-Offboarding - 7 days before last day",
    "description": "Prepare for employee departure",
    "isEnabled": true,
    "isSchedulingEnabled": true,
    "executionConditions": {
        "@odata.type": "#microsoft.graph.identityGovernance.triggerAndScopeBasedConditions",
        "scope": {
            "@odata.type": "#microsoft.graph.identityGovernance.ruleBasedSubjectSet",
            "rule": "(department eq 'Sales')"
        },
        "trigger": {
            "@odata.type": "#microsoft.graph.identityGovernance.timeBasedAttributeTrigger",
            "timeBasedAttribute": "employeeLeaveDateTime",
            "offsetInDays": -7
        }
    },
    "tasks": [
        {
            "displayName": "Remove from all groups",
            "taskDefinitionId": "1953a66c-751c-45e5-8bfe-01462c70da3c",
            "isEnabled": true
        },
        {
            "displayName": "Remove all license assignments",
            "taskDefinitionId": "8fa97d28-3e52-4985-b3a9-a1126f9b8b4e",
            "isEnabled": true
        },
        {
            "displayName": "Send email to manager",
            "taskDefinitionId": "aab41899-9972-422a-9d97-f626014578b7",
            "isEnabled": true,
            "arguments": [
                {
                    "name": "subject",
                    "value": "Employee {{userDisplayName}} leaving in 7 days"
                }
            ]
        }
    ]
}

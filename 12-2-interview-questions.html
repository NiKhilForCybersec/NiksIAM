L1 - Fundamentals Questions

Answer: Authentication (AuthN) verifies WHO you are - it confirms your identity through credentials like passwords, biometrics, or tokens. Authorization (AuthZ) determines WHAT you can do - it controls access to resources based on your permissions, roles, or policies.

Think of it like an airport: showing your ID is authentication, while your boarding pass (authorizing you to board a specific flight) is authorization. Authentication always happens first; you must prove your identity before the system can determine your permissions.

HTTP Status Codes: 401 Unauthorized = authentication failure, 403 Forbidden = authorization failure.

Answer:

  • Something you know: Passwords, PINs, security questions
  • Something you have: Hardware tokens, smart cards, mobile phones, security keys
  • Something you are: Biometrics - fingerprints, facial recognition, iris scans

MFA (Multi-Factor Authentication) requires two or more factors from different categories. Using a password AND security question is NOT true MFA since both are "something you know."

Answer: The principle of least privilege states that users, systems, and processes should have only the minimum level of access necessary to perform their job functions - no more, no less.

Why it matters:

  • Limits blast radius if an account is compromised
  • Reduces risk of accidental damage from authorized users
  • Supports compliance requirements (SOX, HIPAA, etc.)
  • Makes audit and forensics easier

Implementation: Start with zero access and add only what's needed. Regular access reviews remove accumulated privileges. Just-in-Time (JIT) access grants temporary elevation only when needed.

Answer: Single Sign-On (SSO) allows users to authenticate once and access multiple applications without re-entering credentials.

Benefits:

  • User experience: One password, no fatigue, faster access
  • Security: Fewer passwords to manage/steal, centralized authentication, easier MFA enforcement
  • IT efficiency: Reduced help desk tickets, centralized access management
  • Compliance: Centralized audit trail, consistent policy enforcement

Protocols: SAML 2.0 (enterprise), OIDC (modern/mobile), WS-Federation (Microsoft legacy)

RBAC (Role-Based Access Control):

  • Access based on predefined roles (HR_Manager, Developer, etc.)
  • Users assigned to roles, roles have permissions
  • Simple to understand and audit
  • Best for stable organizations with clear job functions

ABAC (Attribute-Based Access Control):

  • Access based on attributes (user, resource, environment, action)
  • Dynamic, context-aware decisions
  • More flexible but complex to implement
  • Best for dynamic, fine-grained access requirements

When to use: Many organizations use hybrid RBAC+ABAC - roles for base access, attributes for fine-grained control.

L2 - Technical Questions

SP-Initiated Flow (most common):

  1. User attempts to access Service Provider (SP) application
  2. SP generates SAML AuthnRequest, redirects user to Identity Provider (IdP)
  3. IdP authenticates user (if not already authenticated)
  4. IdP generates SAML Response containing Assertion with user attributes
  5. User is redirected back to SP with signed SAML Response
  6. SP validates signature, extracts attributes, creates session

Key components: Assertion (claims about user), Bindings (HTTP-POST, HTTP-Redirect), Profiles (Web Browser SSO)

OAuth 2.0: An authorization framework that allows third-party applications to access resources on behalf of a user without sharing credentials. It issues access tokens for API access but does NOT authenticate users.

OpenID Connect (OIDC): An authentication layer built on top of OAuth 2.0. It adds ID tokens (JWTs) that contain user identity claims, a UserInfo endpoint, and standardized scopes (openid, profile, email).

Key difference: OAuth answers "what can this app access?" OIDC answers "who is this user?"

Grant types: Authorization Code (+ PKCE for public clients), Client Credentials (machine-to-machine), Device Flow (IoT), Implicit (deprecated)

Step-by-step flow:

  1. AS-REQ: Client sends username + timestamp (encrypted with user's password hash) to KDC
  2. AS-REP: KDC returns TGT (encrypted with krbtgt hash) + session key (encrypted with user's hash)
  3. TGS-REQ: Client requests service ticket using TGT + authenticator
  4. TGS-REP: KDC returns service ticket (encrypted with service's hash)
  5. AP-REQ: Client presents service ticket to target service

Key security: Password never sent over network. TGT proves authentication without re-entering password. Only KDC can create valid TGTs (krbtgt key).

Definition: Conditional Access policies evaluate signals about the user, device, location, and application to make real-time access decisions. It's the core of Zero Trust "never trust, always verify."

Common conditions:

  • User/Group membership
  • Device compliance state
  • Location (named locations, countries)
  • Application being accessed
  • Sign-in risk level
  • Device platform

Common controls: Block access, Require MFA, Require compliant device, Require approved app, Session controls (sign-in frequency, app restrictions)

Best practice: Start with report-only mode, test with What-If tool, create baseline policies for all users, emergency break-glass accounts excluded.

Key principles:

  • Single source of truth: Use one IdP (Okta, Entra ID, Ping) as the authoritative identity source
  • Federate to clouds: SAML/OIDC federation from central IdP to AWS, Azure, GCP
  • Unified RBAC model: Map consistent roles across cloud providers
  • Centralized governance: IGA platform for access reviews across all clouds
  • Workload identity: Use cloud-native identity (AWS IAM roles, Azure Managed Identity, GCP service accounts) for machine-to-machine

Tools: AWS IAM Identity Center, Azure RBAC, GCP IAM, CIEM solutions for permission analytics

L3 - Expert Questions

Attack: An attacker who extracts the krbtgt account's NTLM hash can forge Ticket Granting Tickets (TGTs) for any user, with any group memberships, including non-existent users. This provides complete domain persistence.

Detection:

  • TGT lifetime longer than domain policy
  • TGT for user who never performed AS-REQ
  • Impossible ticket timestamps
  • Tickets with SID history for non-existent users

Prevention/Remediation:

  • Reset krbtgt password TWICE (allows current tickets to expire)
  • Protect Domain Controller and krbtgt account
  • Enable Credential Guard
  • Use Protected Users group for privileged accounts
  • Monitor Event ID 4768, 4769 for anomalies

Core principles:

  • Verify explicitly: Authenticate and authorize based on all available data points
  • Use least privilege: JIT/JEA access, risk-based adaptive policies
  • Assume breach: Segment access, encrypt all traffic, continuous monitoring

Architecture components:

  • Identity: Strong MFA (phishing-resistant), passwordless, device identity
  • Endpoints: Device compliance, EDR integration, device trust levels
  • Applications: Per-app access policies, ZTNA replacing VPN
  • Data: Data classification, DLP, encryption
  • Infrastructure: Micro-segmentation, PAM for admin access
  • Network: Software-defined perimeter, identity-aware proxy

Signals for policy: User identity, device health, location, application sensitivity, data classification, user behavior (UEBA), real-time risk score

Phase 1 - Assessment:

  • Inventory all applications and authentication methods
  • Identify AD dependencies (Kerberos, NTLM, LDAP binds)
  • Map users, groups, OUs to cloud model
  • Assess password policies and compliance requirements

Phase 2 - Coexistence:

  • Deploy hybrid identity (Azure AD Connect or equivalent)
  • Enable Password Hash Sync or Pass-Through Auth
  • Implement Seamless SSO for transparent experience
  • Begin federating cloud-ready applications

Phase 3 - Migration Waves:

  • Migrate applications in waves (cloud-native first)
  • Convert legacy apps to SAML/OIDC where possible
  • Use application proxies for legacy on-prem apps

Phase 4 - Cutover:

  • Move authentication authority to cloud
  • Implement cloud-native MFA and Conditional Access
  • Decommission on-prem infrastructure

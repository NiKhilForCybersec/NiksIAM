Understanding the JML Process

JML (Joiner-Mover-Leaver) is the industry term for managing the complete lifecycle of an identity. In reality, this is where 70% of your IAM work happens - getting the right people the right access at the right time, and removing it when they no longer need it.

HR System Integration: The Authoritative Source

Why HR is the Source of Truth

In properly designed IAM systems, HR is the authoritative source for identity data. This means HR data drives identity creation, not IT. Here's why:

  • Legal employment relationship: HR knows who is legally employed, when they start/end
  • Organizational structure: HR maintains the org chart, reporting relationships, cost centers
  • Job information: HR defines roles, titles, departments that drive access needs
  • Audit trail: HR systems have documented approval for hiring/termination decisions

Birthright Access: What Gets Provisioned Automatically

Birthright access is access automatically granted based on who someone is (role, department, location) without requiring a request. This is the foundation of efficient JML processes.

Example Birthright Rules (SailPoint) 
# Birthright Rules - What everyone gets
IF employee_type = "FTE" THEN
  ASSIGN:
    - AD Group: "All Employees"
    - AD Group: "Domain Users"
    - M365 License: "E3"
    - Okta App: "Email"
    - Okta App: "Teams"
    - Okta App: "SharePoint"

# Department-based birthright
IF department = "Engineering" AND employee_type = "FTE" THEN
  ASSIGN:
    - AD Group: "Engineering-Staff"
    - Okta App: "GitHub"
    - Okta App: "Jira"
    - Okta App: "Confluence"
    - Okta App: "AWS SSO" (read-only role)

# Location-based birthright
IF location = "Toronto" THEN
  ASSIGN:
    - AD Group: "Toronto-Building-Access"
    - AD OU: "OU=Toronto,OU=Users,DC=corp,DC=company,DC=com"

# Job title-based birthright
IF job_title CONTAINS "Manager" THEN
  ASSIGN:
    - AD Group: "Managers"
    - Okta App: "Workday-Manager"
    - Okta App: "TimeTracking-Approver"
ğŸ’¡ Best Practice: Birthright vs Request-Based Access

Birthright (Automatic):

  • Email, collaboration tools (Teams, Slack)
  • Department-level file shares
  • Standard productivity apps
  • Building access appropriate for location

Request-Based (Requires Approval):

  • Sensitive applications (HR data, financial systems)
  • Admin/privileged access
  • Cross-department resources
  • Production environment access

Mover Process: Role Changes & Transfers

The "Mover" process handles internal changes: promotions, transfers, department changes, manager changes. This is where privilege creep happens if not managed properly.

ğŸ”´ The Privilege Creep Problem

Without proper mover processes, employees accumulate access as they move through the organization. A 10-year employee who has been in 4 departments may have access to all 4 department's sensitive data - a major security and compliance risk.

Example: Sarah started in Sales, moved to Marketing, then to Product. Without proper access revocation, she still has access to Sales CRM, Marketing analytics, AND Product roadmaps.

Types of Mover Events

Event Type What Changes IAM Actions Required Risk Level
Department Transfer Department, Manager, possibly location Revoke old dept access, grant new dept birthright, update OU, update groups ğŸ”´ High
Promotion (same dept) Job title, possibly manager May gain additional access (e.g., now a manager), update title attribute ğŸŸ¡ Medium
Manager Change Reporting relationship only Update manager attribute in AD, may affect delegated access ğŸŸ¢ Low
Location Change Office location, possibly time zone Update building access groups, update location attribute ğŸŸ¢ Low
Employee to Contractor Employment type Major access review, remove FTE-specific access, add contractor restrictions ğŸ”´ High
Contractor to Employee Employment type Add FTE birthright access, remove contractor restrictions ğŸŸ¡ Medium

ğŸ’¡ Best Practice: Grace Period for Movers

Many organizations implement a 7-14 day "grace period" where old access is maintained alongside new access. This allows knowledge transfer and prevents productivity loss.

Implementation:

  • Day 1: New access granted
  • Days 1-7: Both old and new access active
  • Day 8: Old access automatically revoked
  • Email reminder on Day 5: "Your Engineering access will be removed in 2 days"

Immediate Offboarding Checklist (First 15 Minutes)

ğŸ”´ CRITICAL: Immediate Actions for Involuntary Termination

These actions must be completed BEFORE or AS the employee is notified:

  1. Disable AD Account 
    Disable-ADAccount -Identity jsmith
  2. Revoke All Active Sessions 
    # Azure AD / Entra ID
Revoke-AzureADUserAllRefreshToken -ObjectId (Get-AzureADUser -ObjectId jsmith@company.com).ObjectId

# Microsoft Graph (newer)
Invoke-MgGraphRequest -Method POST -Uri "https://graph.microsoft.com/v1.0/users/jsmith@company.com/revokeSignInSessions"
  3. Reset Password (prevents cached credential use) 
    Set-ADAccountPassword -Identity jsmith -Reset -NewPassword (ConvertTo-SecureString ([System.Web.Security.Membership]::GeneratePassword(16,4)) -AsPlainText -Force)
  4. Disable in Okta (if separate from AD) 
    # Okta API
POST https://company.okta.com/api/v1/users/00u1abcd/lifecycle/suspend
  5. Disable VPN/Remote Access - Contact network team or disable in VPN admin
  6. Disable Badge Access - Contact physical security

Same-Day Offboarding Actions

  • Convert mailbox to shared (if needed for business continuity)
  • Set up auto-reply if appropriate: "John is no longer with the company. Please contact..."
  • Forward email to manager (with manager approval, time-limited)
  • Remove from distribution lists
  • Remove from Teams/Slack channels

Set-Mailbox -Identity jsmith@company.com -Type Shared

# Set forwarding
Set-Mailbox -Identity jsmith@company.com -ForwardingAddress manager@company.com -DeliverToMailboxAndForward $false
  • Transfer OneDrive ownership to manager
  • Transfer SharePoint site ownership for any sites they owned
  • Reassign shared documents
  • Archive user's data per retention policy
# OneDrive ownership transfer (SharePoint Online Management Shell)
Set-SPOUser -Site https://company-my.sharepoint.com/personal/jsmith_company_com -LoginName manager@company.com -IsSiteCollectionAdmin $true

If the user had privileged access:

  • Force rotation of any shared privileged passwords they knew
  • Review and rotate service account credentials they managed
  • Audit recent privileged actions in PAM system
  • Remove from PAM groups immediately
  • Revoke any active privileged sessions

Disable/remove from all SaaS applications:

  • Salesforce - deactivate user
  • Slack - deactivate (keeps message history)
  • GitHub - remove from org or downgrade to outside collaborator
  • AWS - remove IAM user, remove from SSO assignments
  • Jira/Confluence - deactivate user
  • Any application with direct accounts (not SSO)

Note: If using Okta/Entra SSO with provisioning, deactivating the user in IdP should automatically deprovision from connected apps.
